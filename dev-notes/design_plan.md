良い方針です。よりシンプルな「ワンパス（一括）ビルド」へ刷新するにあたり、追加で検討すべき改善案をまとめます。

一括レンダリングの順序は「最終順」を推奨
提案: Vivliostyle CLI の一括生成を最終順そのもの（00,01,02,03,11-98,99）で行い、PDF後段のページ並べ替えを不要化。
理由: 一括生成後にPDFページを入れ替えると、ページ番号の再計算・目次/しおりの検算が必要。最終順でレンダリングすれば、アウトライン・リンク整合性が自然に保てます。
並列化は HTML 生成（pre/convert/post）側で行い、PDFは最終順で一回生成するのが安定。
セクション境界の明示マーカーで後処理を安定化
既存の「VS-CHAPTER: 」を拡張し、全セクション（00/01/02/03/11-98/99）に「VS-SECTION: 」不可視テキストを先頭注入。
用途:
最終PDFのページ範囲特定（空白挿入・ラベル付与・アウトライン付与・検証）。
keep（chapters subset）時の境界検出にも再利用可。
実装ポイント: post_process に注入ロジックを追加して統一運用。
空白ページ挿入は HTML ではなく PDF 後処理で統一
右ページ開始や奥付の偶数開始を、PDF段階で HexaPDF による「blank_page.pdf」挿入に一本化（現行Step 8/10と同様の手法を汎用関数化）。
Guard HTML（90-appendices-guard.html）は不要化可能。
ページラベル・アウトラインは「最終合成後」に一括適用
ページラベル: 02-03 をローマ小、その他は算用で HexaPDF::Type::NumberTree による PageLabels を最終PDFに設定。
アウトライン: 
add_outline_from_headings!
 を「最終合成後」の最終PDFに対して実行。見出し抽出は data-heading 優先の現行実装でOK。
メリット: 並べ替えや空白挿入後のページ番号を見たうえで確定できるため、再計算や破綻が起きにくい。
keep（chapters subset）対応の仕様整理
一括順序は常に 00,01,02,03,11-98,99 だが、keep で本文の一部だけ含めるケースに対応。
02/03 を含めるかはポリシー化が必要（例: 常に含める/keep しだい）。最終順の「枠」は保ちつつ、マニフェスト生成時に entries をフィルタ。
目次は含める前提なら、TOCのリンク整合も最終PDF後処理でアウトラインを付ければ安全。
並列化の最適点を HTML 側に寄せる
「一括PDFは1回」なので、スループットは Step 5（章HTML生成）とリソース準備で稼ぐ。
VIVLIO_BUILD_CONCURRENCY のデフォルトをマシンコア数ベースに自動推奨（例: min(4, n_cores)）。実行時に検出・ログ出力。
画像最適化は先に済ませておく（Step 1）ことで PDF レンダリングの負担を低減。
Vivliostyle 設定の動的生成を前提に
一括エントリ（entry: []）を動的に生成するヘルパを用意（final manifest）。title/author/size は book.yml から。
章ごと entries ではなく、最終順 entries で 1 回の CLI 実行。--port は固定でOK（単回起動）。
段階的移行の安全策
フラグスイッチ: --onepass（既定） / --legacy を用意。
ロールバック容易性: 実行ログに one-pass のマニフェスト（最終順リスト）と各セクションページ範囲を出力しておく（不具合時の調査が容易）。
自動検証
ビルド後にチェック:
セクション境界のページ奇偶（00/01/02/03/11-98/99 の開始ページが意図どおりか）
PageLabels の整合（02-03 が i.. / 本文が 1.. ）
アウトライン階層（h1..h3 まで、章ごとに範囲内検出）をサマリー出力
失敗時は警告ログのみ（ビルドは継続）にして運用への影響を抑制。
余剰ステップの削減
先行ビルド（02）と付録 merge を廃止。
CSS 仮想連番（Step 2/11）も、可能なら廃止方向へ（最終順一括＋カウンタCSSで代替）。互換のため当面は残しつつ、--no-virtual-css で無効化テスト可能に。
ベンチマークの内蔵
one-pass の timings を自動収集（レンダリング時間・最終処理時間・合計）。
timings_summary.md
 に one-pass 専用節を追加。
並列度（HTML側）と画像重量（opt preset）での感度を記録し、最適点の指針を残す。
この刷新方針に沿って、以下の順で実装を進めるのが安全です。

最終順マニフェストの生成（keep 対応含む）と、一括 Vivliostyle 実行（one_pass_build_all!）

セクション境界マーカーの統一（VS-SECTION）と検証出力
互換モードスイッチの実装（--legacy）
移行完了後にガードHTMLや先行ビルド/付録mergeの残滓を整理
