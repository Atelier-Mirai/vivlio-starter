# ビルド高速化 実験記録（2025-09-07）

本ドキュメントは、Vivlio Starter のビルド高速化に関する実験結果の記録です。対象は Step 3/7/9 の所要時間、および front/tail ページ（00/01, 98/99）の結合ビルドの効果、entries 形式の違いによる所要時間差です。

## 実験環境
- プロジェクト: `vivlio-starter`
- コマンドは原則 `--no-compress --no-clean --log`（ログは info 相当）
- 並列実験時は `VIVLIO_PDF_CONCURRENCY=8`

## 実験1: Step 7 の方式比較（全体ビルド）

計測コマンドと結果（抜粋、Step別タイム）:

### 1) 従来（個別HTML→entries.js→PDF→分割）
```
bin/vs build --no-compress --no-clean --log
== Build Step Timings ==
  - Step 3 (preface prebuild)           18.42s
  - Step 6 (generate toc and pdf)       24.88s
  - Step 7 (build overall pdf and split)  24.55s
  - Step 9 (build front pages and tail)  77.69s
  = TOTAL                              150.82s
```

### 2) 単一HTMLに結合してからPDF（--single_html）
```
bin/vs build --no-compress --no-clean --single_html --log
== Build Step Timings ==
  - Step 3 (preface prebuild)           15.76s
  - Step 6 (generate toc and pdf)       26.22s
  - Step 7 (build overall pdf and split)  21.56s
  - Step 9 (build front pages and tail)  72.36s
  = TOTAL                              141.32s
```

### 3) 章PDF並列生成＋結合（--parallel_pdf, VIVLIO_PDF_CONCURRENCY=8）
```
VIVLIO_PDF_CONCURRENCY=8 bin/vs build --no-compress --no-clean --parallel_pdf --log
== Build Step Timings ==
  - Step 3 (preface prebuild)           26.93s
  - Step 6 (generate toc and pdf)       19.66s
  - Step 7 (build overall pdf and split)  11.04s
  - Step 9 (build front pages and tail)  84.19s
  = TOTAL                              146.29s
```

所見:
- Step 7 の高速化は **並列 > 単一HTML > 従来** の順で効果あり。
- ただし全体では **Step 9** と **Step 3** がボトルネック（Vivliostyle CLI 起動回数の影響が大）。

## 実験2: front/tail（00/01, 98/99）の結合ビルド効果

単一HTMLへ結合して 1 回起動で PDF 化した場合の測定（mac の `/usr/bin/time -p` 計測）。

- 00+01（`combined_00_01.html`）
  - 単一起動: `real 14.33s`
  - 参考（個別の pdf 段階合計）: `12.54s + 12.40s = 24.94s`
  - 差分: 約 **-10.6s**

- 98+99（`combined_98_99.html`）
  - 単一起動: `real 13.74s`
  - 参考（個別の pdf 段階合計）: `18.42s + 17.53s = 35.95s`
  - 差分: 約 **-22.2s**

所見:
- front/tail を結合して **Vivliostyle 起動を 1 回に集約**すると、起動コスト分が大きく短縮。
- 98/99 の方が効果が大きい。

## 実験3: entries 形式の差（Separate vs Combined）

ベンチスクリプト: `bin/bench_entries.rb`
- ベンチ用設定 `vivliostyle.config.bench.js` / `entries_bench.js` を生成し、
  - Case A: Separate（`98-postface.html`, `99-colophon.html`）
  - Case B: Combined（`combined_98_99.html`）
  を各 3 回実行。`/usr/bin/time -p` の `real` を平均化。

実行結果:
```
Case A: Separate (98-postface + 99-colophon)
  Runs: 13.84s / 13.10s / 14.06s → 平均 13.67s

Case B: Combined (combined_98_99.html)
  Runs: 20.51s / 11.31s / 10.90s → 平均 14.24s
```

所見:
- 平均では Separate が僅差で速いが、Combined の **ウォーム状態（2回目以降）** は明確に速い（11.31s, 10.90s）。
- 初回（20.51s）は Vivliostyle/Chromium の **コールドスタート**によるブレが大きい。
- 実運用で連続ビルドや Step 9 の**バッチ化**を行うなら Combined が有利。

## まとめと次の方針
- ボトルネックは **Step 9**（front/tail の個別 PDF 化 ×4）と **Step 3**（preface 先行の単独 PDF 化）。
- 改善施策（案）:
  1. Step 3 のスキップオプション追加（例: `--no-preface-prebuild`）。
  2. Step 9 の **結合ビルド**オプション追加（00+01 と 98+99 をそれぞれ 1 回起動で PDF 化 → 必要に応じて HexaPDF で分割）。
  3. Step 9 内訳のタイマー出力（00/01/98/99 の所要時間を個別に記録）。

## 付記（再現手順）
- 単一HTMLの生成例:
  - `combined_00_01.html` / `combined_98_99.html` は Ruby ワンライナーで `<body>…</body>` 内を抽出して `<section>` として連結。
- entries 形式ベンチの実行:
  - `ruby bin/bench_entries.rb`
