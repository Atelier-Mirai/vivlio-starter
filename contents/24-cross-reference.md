# クロスリファレンス（相互参照）の使い方

:::{.chapter-lead}
本章では、本書で定義している **図・表・リストのナンバリングと相互参照（クロスリファレンス）** の
書き方を説明します。

「この図を参照してください」「詳しくは前の表を参照してください」といった記述を、章番号付きの
番号（図 3-2, 表 4-1 など）とリンク付きで自動生成するための仕組みです。
:::

## 基本の考え方

クロスリファレンスは、次の二段階で構成されています。

1. **定義側**: 図・表・リストに「ラベルID（`@id`）」を付けてキャプションを記述する
2. **参照側**: 本文中から `@id` を使ってその要素を参照する

ビルド時に、各章の図・表・リストに「章番号＋連番」の番号が自動で割り振られ、
本文中の `@id` は「図 3-2」「表 4-1」「リスト 5-3」のような番号付きのテキスト（＋リンク）に
置き換えられます。

## ラベル付きキャプションの書き方（定義側）

### 共通ルール

図・表・リストには、次のような **キャプション行** を付けます。

- 行頭に太字 (`** ... **`) でタイトルを書く
- タイトルの末尾に `@ラベルID` を付ける
- 直後に続くブロック要素（コードブロック / テーブル / 画像）がその対象になります

ラベルIDは、本文中で参照するときに使う名前です（例: `@css-table-rotate`）。

### コード（リスト）の例

CSS コード例を「リスト」として扱い、相互参照したい場合は次のように書きます。

````markdown
** 回転テーブルのCSS定義 @css-table-rotate **
```css:table.css
table {
  border-collapse: collapse;
  /* 略 */
}
```
````

- キャプション行: `** 回転テーブルのCSS定義 @css-table-rotate **`
- 直後のブロック: ```css:table.css ～ ``` → **コードブロック** なので「リスト」として扱われます。

### 表の例

テーブル（表）をナンバリングの対象にしたいとき:

```markdown
** 回転テーブルのプロパティ一覧 @prop-list **
| プロパティ名 | 役割     |
| :----------- | :------- |
| --scale      | 縮尺率   |
| --shift-y    | Y軸シフト |
```

- キャプション行の直後が Markdown テーブルなので、「表」として扱われます。

### 図の例

画像を図として扱い、参照できるようにしたい場合:

```markdown
** 回転テーブルのイメージ図 @table-rotate-image **
![回転テーブルのイメージ図](table-rotate.png)
```

- 直後が画像（`![]()`）なので、「図」として扱われます。

### 番号を付けない例（ただのコード例など）

`@id` を書かなければ、その要素は **番号も相互参照も付かない単なる例** として扱われます。

````markdown
** CSSの例 **
```css
h1 {
  color: red;
}
```
````

このように書くと、見出しとして太字にはなりますが、

- 「リスト 4-1」などの番号は付かない
- 本文から `@id` で参照することもできない

という扱いになります。

## 本文での参照方法（参照側）

### `@id` による参照

本文からは、定義したラベルIDを **`@id` という形でそのまま書くだけ** で参照できます。

例:

```markdown
横長表の調整方法については、@css-table-rotate を参照してください。
ここで定義されているカスタムプロパティの一覧は、@prop-list にまとめています。
レイアウト全体のイメージについては、@table-rotate-image を参照してください。
```

ビルド結果では、次のような形になります（例）:

- `@css-table-rotate` → 「リスト 4-1」へのリンク
- `@prop-list` → 「表 4-2」へのリンク
- `@table-rotate-image` → 「図 4-3」へのリンク

### 表示されるテキスト

番号部分（リスト 4-1 / 表 3-2 / 図 1-5）は、

- **章番号**（第n章）
- **その章の中での連番**

から自動的に計算されます。著者が手動で番号を管理する必要はありません。

## 手動IDと自動ID（@auto / @omakase）

### 手動ID（@id）

意味のある名前でIDを付けたい場合は、キャプション行に直接 `@id` を書きます。

```markdown
** 回転テーブルのCSS定義 @css-table-rotate **
```

- 本文では `@css-table-rotate` と書くだけで参照できます。
- ID は本書全体で一意である必要があります（同じIDを複数の図表リストに使うことはできません）。

### 自動ID（@auto / @omakase）

ID を自分で考えたくない場合は、キャプション行に `@auto` または `@omakase` を書きます。

````markdown
** 別のコード例 @auto **
```ruby
def prime?(n)
  # ...
end
```
````

- この場合、システムが「リスト 4-4」のような **機械的で一意なID** を内部的に割り当てます。
- 自動IDの実際の名前（例: `@list-4-4`）は、ビルド後の **ID一覧レポート** やビルドログで確認します。

### システム予約ID（予約語）

`@id` という記法は、本章で説明したクロスリファレンスのほかに、`config/post_replace_list.yml` で定義される **ビルド時マクロ** でも使われます。以下の `@id` は **システム予約語** として扱われ、クロスリファレンス側では「未定義のラベルID」として警告されません。

#### 自動採番用の予約ID

| 予約ID | 用途 |
| :--- | :--- |
| `@auto` | 自動ID割り当て（§4.2） |
| `@omakase` | 同上（別名） |
| `@id` | 同上（別名） |

#### `post_replace_list.yml` マクロの予約ID（完全一致）

| 予約ID | マクロの役割 |
| :--- | :--- |
| `@vspace` | 縦方向の余白挿入（`@vspace:1.5lh` / `@vspace:10`（単位省略時は mm）。負値で上に詰める） |
| `@nega` / `@posi` | 上下方向への余白挿入（`@vspace` の旧記法。`@nega:3`, `@posi:5` のように数値を続ける） |
| `@comment` / `@commend` | 編集者コメント囲み（`@comment:...@commend`） |

#### `post_replace_list.yml` 絶対配置マクロの予約接頭辞（**Planned**）

以下の接頭辞に続く数字列（`@lu25`, `@ld30` など）も予約されています。現時点では `post_replace_list.yml` 側でコメントアウトされており、次期リリースで正式サポート予定です（詳細は `CHANGELOG.md` の **Planned > 記法・置換ルール**）。

- `@lu` / `@ld` / `@ru` / `@rd` / `@ur` — L字ガイド線（4 隅＋上右）
- `@ls` / `@rs` — 水平ガイド線（左／右）
- `@us` / `@ds` — 垂直ガイド線（上／下）

:::{.tip}
**著者が新しいラベルIDを付けるときの注意**

上記の予約語と衝突する ID（例: `@div` という図を定義する、`@lu25` というリストIDを付ける等）は使用しないでください。予約語扱いとなるため、本文から参照してもクロスリファレンスとして展開されません。
:::

## IDの一覧と重複について

### ID一覧レポート

ビルド時には、全ての図・表・リストとそのラベルIDをまとめた **ID一覧レポート** を出力します
（例: `Cross Reference Map`）。

このレポートには、少なくとも次の情報が含まれます。

- 章ファイル名（例: `71-install.md`）
- ラベルID（例: `@css-table-rotate`, `@prop-list`, `@table-rotate-image`）
- 種別と番号（リスト/表/図 + `章番号-連番`）
- 手動IDか自動IDか（manual/auto）
- キャプション行の本文（タイトル）

将来的には、`vs xref:list` のようなコマンドで ID 一覧を表示する機能も追加される予定です。

### IDの一意性（重複禁止）

ラベルIDは、リスト・表・図をまたいで **本書全体で一意でなければなりません**。

- 例えば、リストと図の両方で `@css-table-rotate` を定義することはできません。
- 同じIDを持つ定義が複数存在する場合、ビルド時に重複エラーとして検出され、エラーメッセージが表示されます。ビルドは最後まで行われますが、意図したリスト番号等が付けられていない{虞|おそれ}がありますので、原稿を確認してください。
- エラーメッセージには、どのファイルのどの行で重複しているかが表示されます。

意味のあるIDを種別ごとに分けて使いたい場合は、次のようにID自体を変えてください。

- `@css-table-rotate-css` （リスト用）
- `@css-table-rotate-fig` （図用）

このようにしておくと、本文中の参照も明確になり、後から読んだときにも意味が分かりやすくなります。

## ラベルIDの命名規則（推奨）

ラベルID（`@id`）には厳密な制約はありませんが、本書では次のような **分かりやすく衝突しにくい命名** を推奨します。

### 基本ルール

- 文字種は **英小文字 + 数字 + ハイフン** に統一する（例: `sample-code-56`）
- 単語はハイフンでつなぐ（kebab-case）
- 必要に応じて **章番号を末尾に付ける**（例: `-55`, `-56`）

このようにしておくと、IDを見ただけで「どの章の何を指しているか」がある程度分かります。

### 章ごとのサンプルでの例

例えば、クロスリファレンスの動作確認用の章では、次のような ID を付けています。

- `@sample-code-56`  …… 56 章のサンプルコード
- `@data-table-56`   …… 56 章のデータ表
- `@diagram-56`       …… 56 章の図

別の章で同様のテストを増やしたい場合は、次のように章番号だけ変えると分かりやすくなります。

- `@sample-code-55`, `@sample-code-56`
- `@diagram-11`, `@diagram-12`, `@diagram-13`

### 種別ごとに意味のある名前を付ける

意味が分かりやすい ID を付けたい場合は、用途に応じて単語を組み合わせます。

- コード例: `@ruby-sample`, `@js-sample`, `@js-sample2`, `@prime2`, `@prime3`
- 表: `@basic-table`, `@data-table-56`
- 図: `@einstein`, `@diagram-56`

どの種別（リスト/表/図）で使う ID かは、**キャプション行の直後のブロックの種類** から自動で判定されます。ID そのものに `fig-` や `table-` などの接頭辞を付けるかどうかは任意ですが、

- その章で図が多い場合は `@diagram-...`
- 表が多い場合は `@data-table-...`

のようにしておくと、後から見返したときにも意味が取りやすくなります。

### 自動IDとの付き合い方

`@auto` / `@omakase` を使った場合、内部的には `@list-2-1` や `@fig-2-3` のような ID が自動生成されます。

- これらの ID は **ツール側で一意になるように管理される内部用 ID** と考えてください。
- 本文から参照するときは、なるべく **自分で付けた意味のある ID**（`@ruby-sample` など）を使うことを推奨します。

このようなルールをゆるやかに守っておくと、章が増えたあとでも ID の重複や混乱を防ぎやすくなります。

## トラブルシューティング

### よくある問題

#### IDが見つからない
- 原因: タイプミス、または定義前に参照している
- 解決: ID一覧レポートで正しいスペルを確認

#### 重複エラー
- 原因: 同じIDを複数の要素で使用
- 解決: IDを一意になるように変更

#### 番号がおかしい
- 原因: ビルドのキャッシュ残留
- 解決: `vs clean` でクリーンアップ

### エラーメッセージの例

`vs preflight` や `vs build` で表示される主なエラーメッセージと対処法です。

#### ラベルIDの重複

```
🔴 25-querystream.md:25 - ラベルID '画像(左寄せ) @img-left' は重複しています
        重複箇所: 25-querystream.md: 25, 30
```

同じ `@id` が複数箇所で定義されています。IDは書籍全体で一意にする必要があります。いずれかのIDを変更してください。

#### 孤立ラベル（未参照の警告）

```
🟡 24-cross-reference.md:329 - 孤立ラベル 'Prime @prime' は未参照です
```

定義されたラベルが本文中のどこからも `@prime` で参照されていません。参照を追加するか、不要であればラベル定義を削除してください。これは警告（🟡）であり、ビルドは正常に完了します。

### デバッグ方法

1. **個別ビルド**: `vs build [章番号]` で特定章のみテスト
2. **バリデーション**: `vs lint` で基本的な問題を検出

## まとめ

クロスリファレンス機能を使いこなすことで：

- **参照の一貫性**: 章番号付きで正確な参照
- **リンク機能**: PDF内でジャンプ可能な相互参照
- **自動管理**: 番号付けの自動化と重複防止
- **柔軟性**: 手動IDと自動IDの使い分け

これらの機能を活用して、読者にとって使いやすく、著者にとって管理しやすい技術書を作成しましょう！
