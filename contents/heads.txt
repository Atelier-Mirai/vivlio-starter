# はじめに

ようこそ、Vivlio Starter の世界へ！

「自分の本を作ってみたい」——そう思ったことはありませんか。日々の仕事で培った技術的な知見、趣味で深めた専門知識、あるいは誰かに伝えたい物語。頭の中にある「伝えたいこと」を一冊の本にまとめ、手に取れる形にする。それは、とても{愉|たの}しく、やりがいのある営みです。

しかし、いざ書籍を作ろうとすると、執筆以外の作業に多くの時間を取られてしまいます。ページレイアウト、目次の生成、画像の配置、表紙の作成、入稿用 PDF の調整……。Vivlio Starter は、そうした煩雑な作業をすべて引き受け、書き手が「書くこと」に集中できる環境を提供します。

:::{.chapter-lead}
Vivlio Starter は Markdown で書いた原稿から高品質な PDF・EPUB を生成する書籍制作システムです。CSS 組版エンジン Vivliostyle をコアに据え、原稿の前処理から目次生成、表紙作成、トンボ付き入稿用 PDF の出力まで、執筆から入稿に至るすべての工程を自動化します。この本自体が Vivlio Starter で作られた実践的な記述例でもあります。
:::

## Vivlio Starter でできること

- **Markdown で執筆** — 使い慣れた記法でさくさく書ける。特別なフォーマットを覚える必要はありません
- **コマンド一発でビルド** — `vs build` ひとつで、原稿が美しい PDF に変わります
- **印刷入稿に対応** — トンボ・塗り足し付きの PDF を生成。印刷所にそのまま入稿できます
- **電子書籍も出力** — EPUB 形式での出力にも対応。電子出版の道も開かれています
- **テーマで簡単デザイン** — `book.yml` でアクセントカラーや扉絵を選ぶだけで、統一感あるデザインに
- **環境構築も自動** — `vs new` でプロジェクトを作成し、`vs doctor` で必要なツールを自動セットアップ

Markdown は、ジョン・グルーバー（John Gruber）とアーロン・スワーツ（Aaron Swartz）によって2004年に開発された軽量マークアップ言語です。「書きやすく、読みやすい」ことを目指して設計され、プレーンテキストで構造化された文書を作成できます。

### 開発者と歴史

#### 開発者
- **ジョン・グルーバー**: Daring Fireball の著者、Markdown の考案者
- **アーロン・スワーツ**: プログラマー、情報活動家。14歳で RSS 1.0 の仕様策定に貢献

#### 歴史的背景

=== 18-lint.md ===
# Textlint

:::{.chapter-lead}
Vivlio Starter には、Markdown で書かれた文章の品質をチェックする textlint 機能が統合されています。表記揺れ、冗長表現、文体の混在などを自動検出し、執筆品質の向上をサポートします。
:::

## textlint とは

:::{.section-lead}
textlint は、自然言語の文章を検査するための Linter ツールです。プログラムコードの品質チェックツール（ESLint や RuboCop など）の文章版と考えるとわかりやすいでしょう。
:::

textlint は Node.js で動作するオープンソースツールで、特に日本語の技術文書に対して強力な校正機能を提供します。Vivlio Starter では、この textlint を CLI コマンドとして統合し、`contents/` ディレクトリ内の Markdown ファイルを簡単にチェックできるようにしています。

### textlint の主な機能

- **表記揺れの検出**: 「ユーザ」と「ユーザー」、「javascript」と「JavaScript」などの表記の不統一を検出
- **文体の統一**: 「です・ます調」と「だ・である調」の混在をチェック
- **冗長表現の検出**: 助詞の重複、接続詞の連続使用などを指摘
- **技術用語のスペルチェック**: IT 用語の誤記を検出

# 扉絵と装飾画像

:::{.chapter-lead}
Vivlio Starter では、章の扉ページに表示する背景画像（frontispiece）と、節見出しの装飾画像（ornament）を設定できます。画像を使った華やかなデザインと、シンプルで洗練されたデザインの2つのスタイルから選択可能です。
:::

## frontispiece と ornament とは

:::{.section-lead}
frontispiece は章の扉ページに表示される縦長の背景画像、ornament は節見出しに表示される横長の装飾画像です。これらを設定することで、書籍全体に統一感のあるビジュアルデザインを実現できます。
:::

### frontispiece（扉絵）

frontispiece は、各章の扉ページ（章タイトルが表示されるページ）の背景に配置される画像です。縦長（portrait）の画像が使用され、章の雰囲気を演出します。

- **推奨アスペクト比**: 4:3 または √2:1（A4縦）
- **推奨サイズ**: 幅 2880px 程度
- **用途**: 章扉ページの背景画像

# ビルド（vs build）

:::{.chapter-lead}
原稿の執筆が一段落したら、`vs build` コマンドで書籍を組版しましょう。閲覧用 PDF、印刷入稿用 PDF、電子書籍（EPUB）の三つの形式を、ひとつのコマンドで生成できます。

`vs build` は Vivlio Starter の中核となるコマンドです。原稿の前処理から画像最適化、Markdown→HTML 変換、Vivliostyle による組版、PDF 結合、アウトライン付与まで、書籍制作に必要な一連の工程を自動的に実行します。

## はじめてのビルド

:::{.section-lead}
まずは最も基本的な使い方から見ていきましょう。
:::

```bash
vs build
```

引数なしで実行すると、`config/catalog.yml` に定義された全章を対象に**フルビルド**を行います。出力される形式は `config/book.yml` の `output.targets` で決まります。

=== 21-metrics.md ===
# Metrics

:::{.chapter-lead}
Vivlio Starter には、Markdown で書かれた原稿を定量的に分析する metrics 機能が統合されています。文章の「難易度」「表現の多様性」「構成のバランス」を客観的に把握し、より読みやすい技術書の執筆をサポートします。
:::

## metrics とは

:::{.section-lead}
metrics は、執筆中の原稿を統計的に分析し、文章品質を数値化するツールです。漢字比率、語彙の豊かさ、読解難度などを自動計算し、改善のヒントを提供します。
:::

技術書の執筆では、内容の正確さだけでなく、読みやすさも重要です。metrics を活用することで、以下のような客観的な指標を得られます。

### metrics の主な機能

- **基本統計**: 文字数、行数、文の数、節の数を集計
- **語彙難度**: 漢字比率と平均語長から文章の「硬さ」を測定
- **語彙多様度**: TTR（Type-Token Ratio）で表現の豊かさを評価
- **読解難度**: 日本語版 Flesch スコアで読みやすさを判定

=== 22-pdf-read.md ===
# PDF 読み取りコマンドの使い方

:::{.chapter-lead}
`vs pdf:read` は PDF ファイルからテキストと画像を抽出し、Vivlio Starter の原稿形式（Markdown）に変換するコマンドです。既存の書籍 PDF や配布資料を執筆素材として再利用したい場合に活用できます。
:::

---

## 1. 概要

`vs pdf:read` は **Standard Mode** と **Enhanced Mode** の 2 段階で機能を提供します。

| 項目 | Standard Mode | Enhanced Mode |
| --- | --- | --- |
| ライセンス | MIT | AGPL-3.0 |
| 変換対象 | テキストのみ | テキスト + 画像 + OCR |
| 依存ライブラリ | PDF::Reader | HexaPDF, ruby-vips, Tesseract |
| 主な用途 | 参考資料の粗変換 | 出版クオリティの再利用 |

Standard Mode は vivlio-starter 本体に組み込まれており、追加インストール不要で動作します。Enhanced Mode を利用するには `vivlio-starter-pdf` gem が必要です。

=== 23-querystream.md ===
# データ展開機能の使い方

:::{.chapter-lead}
原稿の中に外部データを埋め込む **QueryStream 記法** の使い方を解説します。参考書籍の一覧、都道府県データ、気象観測データなど、繰り返し登場する情報を YAML で一元管理し、テンプレートに従って自動展開できます。データファイルとテンプレートを置くだけで使えるため、設定ファイルの編集は不要です。
:::

---

## 1. 概要

Vivlio Starter のデータ展開機能は、3 つの要素で構成されています。

| 要素 | 置き場所 | 役割 |
| --- | --- | --- |
| データファイル | `data/*.yml` | 展開するデータの実体（YAML 配列） |
| テンプレート | `templates/_[名前].md` | データの表示レイアウト |
| QueryStream 記法 | `contents/*.md` の原稿内 | 展開指示（1 行で完結） |

`vs build` を実行すると、原稿内の QueryStream 記法が自動的にデータとテンプレートを使って展開されます。原稿ファイル（`contents/` 内）は書き換えられません。


=== 31-cross-reference.md ===
# クロスリファレンス（相互参照）の使い方

:::{.chapter-lead}
本章では、本書で定義している **図・表・リストのナンバリングと相互参照（クロスリファレンス）** の
書き方を説明します。

「この図を参照してください」「詳しくは前の表を参照してください」といった記述を、章番号付きの
番号（図 3-2, 表 4-1 など）とリンク付きで自動生成するための仕組みです。
:::

## 1. 基本の考え方

クロスリファレンスは、次の二段階で構成されています。

1. **定義側**: 図・表・リストに「ラベルID（`@id`）」を付けてキャプションを記述する
2. **参照側**: 本文中から `@id` を使ってその要素を参照する

ビルド時に、各章の図・表・リストに「章番号＋連番」の番号が自動で割り振られ、
本文中の `@id` は「図 3-2」「表 4-1」「リスト 5-3」のような番号付きのテキスト（＋リンク）に
置き換えられます。

科学の世界に革命をもたらした{Albert Einstein|アルバート・アインシュタイン}。言葉が遅いと心配された少年が、やがて「20世紀最高の物理学者」と呼ばれるまでに至った背景には、型破りな発想力と、誰もが信じる「常識」に疑問を投げかける勇気がありました。

権威を嫌い、好奇心のままに宇宙の謎を追い求めた生涯——その軌跡をたどることで、わたくし達は「挑戦することの意味」を改めて問い直すことができるでしょう。
:::

---

## 言葉の遅い少年

::: {.section-lead}
1879年3月14日、ドイツ南部の小さな町ウルムに、一人の少年が生まれました。アルバート・アインシュタイン。幼い頃のアルバートは口数が少なく、両親を心配させる子どもでした。しかし、その静けさの奥には、常人には想像もつかない深い思索が宿っていたのです。
:::

# 終わりに

本書を最後までお読みくださり、心より感謝申し上げます。

Markdown で原稿を書き、`vs build` コマンドひとつで美しい書籍が生まれる。その一連の流れを、本書を通じて体験していただけたなら幸いです。技術書、解説書、同人誌、社内マニュアル——Vivlio Starter の世界は、あなたが書きたいものの数だけ広がっています。いろいろと創意工夫しながら、ぜひさまざまな書籍づくりに挑戦してみてください。

---

「福祉」という言葉があります。「福」も「祉[^114]」も、どちらも「めぐみ、さいわい」という意味を持つ美しい言葉です。

[^114]: 「祉い」と書いて「さいわい」と読みます。天からの恵みがその身に止まるという意味です。

わたくし達が身につけた技術を、ただその一身に留めることなく、周りの方々へ伝えてください。原稿を書き、本の形に整え、世に送り出すこと——それは誰かの学びを助け、誰かの視野を広げ、誰かの心に届く営みです。技術を通じて人々に恵みを{齎|もたら}し、幸せを届ける。それもまた、ひとつの福祉のかたちではないでしょうか。

「熱き心、{逞|たくま}しき{腕|かいな}、冷静な頭脳」

学生時代に教わった言葉です。福祉を志す者は、人としての熱い思いと暖かい心を持ち、冷静な判断力を{以|もっ}て力強く行動せよ、と。

執筆活動に{準|なぞら}えるなら——熱き心とは、伝えたいことへの情熱であり、読者の理解を願う誠実さ。{逞|たくま}しき{腕|かいな}とは、原稿と向き合い、書き直し、磨き続ける実践力。そして冷静な頭脳とは、構成を整理し、読者の視点から内容を吟味する思考力。この三つを携えて書かれた言葉は、きっと読み手の心に届くことでしょう。

```
