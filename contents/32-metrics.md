# Metrics

:::{.chapter-lead}
Vivlio Starter には、Markdown で書かれた原稿を定量的に分析する metrics 機能が統合されています。文章の「難易度」「表現の多様性」「構成のバランス」を客観的に把握し、より読みやすい技術書の執筆をサポートします。
:::

## metrics とは

:::{.section-lead}
metrics は、執筆中の原稿を統計的に分析し、文章品質を数値化するツールです。漢字比率、語彙の豊かさ、読解難度などを自動計算し、改善のヒントを提供します。
:::

技術書の執筆では、内容の正確さだけでなく、読みやすさも重要です。metrics を活用することで、以下のような客観的な指標を得られます。

### metrics の主な機能

- **基本統計**: 文字数、行数、文の数、節の数を集計
- **語彙難度**: 漢字比率と平均語長から文章の「硬さ」を測定
- **語彙多様度**: MATTR（移動平均 Type-Token Ratio）で表現の豊かさを評価
- **読解難度**: 建石式リーダビリティー（RS）で読みやすさを判定
- **章別バランス**: 各章の分量を可視化し、構成の偏りを検出

## metrics コマンドの基本

:::{.section-lead}
Vivlio Starter では、`vs metrics` コマンドで原稿の統計分析を実行します。全章の概要表示から、特定章の詳細分析まで、柔軟な使い方が可能です。
:::

### 基本的な使い方

引数なしで実行すると、全章の概要が表示されます。

```bash
vs metrics
```

### 特定の章を分析

章番号を指定すると、その章の節まで詳細に表示されます。

```bash
# 第2章のみ分析
vs metrics 2

# 複数の章を指定
vs metrics 1 3 5
```

### 範囲指定

章番号の範囲を指定して、複数の章をまとめて分析できます。

```bash
# 第1章から第3章まで
vs metrics 1-3

# 範囲と個別指定の組み合わせ
vs metrics 1-3 5 8-10
```

### ゼロ埋め正規化

章番号は自動的にゼロ埋めされます。`1` と入力しても `01` として解釈されます。

```bash
# 以下は同じ意味
vs metrics 1
vs metrics 01
```

## コマンドオプション

:::{.section-lead}
metrics コマンドには、出力内容をカスタマイズするオプションが用意されています。
:::

### --all オプション

全章の節まで詳細に表示します。ライブ表示中もすべての章が展開され、節ごとの分量推移が確認できます。

```bash
vs metrics --all
```

### --warn オプション

警告がある章のみを節まで詳細に表示します。分量が基準から外れている章を素早く特定できます。

```bash
vs metrics --warn
```

### --json / --yaml オプション

統計情報を構造化データとして出力します。章ごとの基本統計に加え、集計済みの `totals` を含むため、外部ツールで再計算する必要がありません。

```bash
vs metrics --json
vs metrics --yaml
```

JSON/YAML 出力時はライブ UI をスキップして即座に結果を返します。キャッシュされた章は再解析せず、トークン頻度マップ（語彙多様度の再集計用）や読解難度の特徴量を含むキャッシュをそのまま利用します。

## 実行結果の見方

:::{.section-lead}
metrics の出力は、基本情報、詳細分析、章別分量の3つのセクションで構成されています。
:::

### 基本情報

文章全体の基礎的な統計を表示します。

```
📊 文章統計 — 基本情報
- 文字数: 12,345 文字（改行除く）
- 行数: 123 行

📊 文章統計 — 文構造
- 文の数: 45 文（平均 274 文字/文）
- 節の数: 134 節（平均 92 文字/節）
- 読点数: 89 個
```

### 詳細分析

語彙難度、語彙多様度、読解難度を分析します。語彙難度には、本文を構成する文字種の割合（漢字・ひらがな・カタカナ・英字・その他）も併記します。

```
📈 詳細分析

【語彙難度】
- 漢字比率: 適切（28.5%） — 理想的な範囲 25〜35%
- 平均語長: 2.3 文字
- 文字種構成: 漢字 28.5% ／ ひらがな 45.2% ／ カタカナ 12.1% ／ 英字 5.7% ／ その他 8.5%

【語彙多様度】
- 使用語彙数: 3,900 語（異なり語）／ 総語数 54,000 語
- 語彙の豊かさ: 標準的（MATTR: 0.58）
  - 評価基準: 0.7 以上=非常に豊富、0.6〜0.7=豊富、0.5〜0.6=標準的、0.5 未満=単調

【読解難度】
- 評価: Standard（一般的なビジネス・実用書）
- スコア: 45.2 点
```

:::{.note}
**コードは分析対象から除外されます。** 漢字比率・語彙多様度・読解難度は、文章としての読みやすさを測る指標です。コードブロック（` ``` ` で囲んだ部分）やインラインコードは、英数字・記号の塊が指標を実態より平易側に歪めるため、これらの分析からは除かれます（章別分量の文字数には、ページを占める実体としてコードも含めます）。
:::

### 章別分量

各章の文字数をバーグラフで可視化します。基準から外れている章には警告が表示されます。

```
📚 章別の分量

第1章 はじめに              [##########  ] 2,500 文字
第2章 基本の実装            [############] 4,800 文字 ⚠️ やや長い
第3章 まとめ                [###         ]   600 文字 ⚠️ 加筆検討
```

### 章間のばらつき

漢字比率と平均文長について、章ごとの値が全体平均からどれだけばらついているかを示します。平均と標準偏差（ばらつきの幅）を表示し、平均から大きく外れた章を「高め／低め（長め／短め）」に分けて挙げます。

```
📐 章間のばらつき（本文の章のみ）

- 漢字比率: 平均 27.3% ／ ばらつき ±4.3
  高め: 第33章 35.2%、第32章 34.4%
  低め: 第51章 17.9%、第12章 18.9%
- 平均文長: 平均 61.5字 ／ ばらつき ±19.0
  長め: 第33章 97.8字、第51章 97.0字、第21章 92.4字
  短め: 第12章 37.5字、第13章 41.3字、第22章 41.8字
```

全体の平均だけでは「どの章が浮いているか」が分かりません。漢字比率が高め＝難しめ、低め＝易しめ、文が長め＝読みにくい可能性など、推敲（書き直し）の手掛かりになります。前書き・付録・後書き（`exclude_chapters` の章）と、本文のない章は比較対象から除き、対象が 2 章以上あるときだけ表示します。

### 見直したい長い文

特に長い文（80 文字以上）の上位を、出現位置（章・行番号）と冒頭の抜粋つきで最大 5 件まで挙げます。

```
📝 見直したい長い文（ワースト5）

1. 第 3章 L274（118字）: 索引候補の抽出では見出し語とその読みを判定し…
2. 第21章 L 88（ 92字）: Markdown では段落の途中で改行しても整形時…
```

行番号があるので、エディタで該当箇所へすぐ移動できます。長い文は一文に情報を詰め込みすぎていることが多く、二文に分ける、接続助詞「〜が、」を切るなどの推敲対象を見つけられます。コードブロックや見出し・表などの構造行は対象から除き、本文の文だけを評価します（文字数はインラインコードを除いた本文基準です）。

### 文末表現のリズム

文末表現の内訳（です・ます／体言止め／だ・である／その他）と、同じ文末が続いて単調になっている箇所を示します。

```
🎶 文末表現のリズム

- 文末の内訳: です・ます 89%／体言止め 1%／だ・である 0%／その他 11%
- 同じ文末の連続（5 つ以上、多い順）:
    第61章 L538 付近（ます。が32連続）
    第42章 L115 付近（ます。が28連続）
    第61章 L220 付近（ます。が17連続）
    （ほか 70 箇所）
```

「です・ます」ばかりが続くと、文章が単調で平板な印象になります。内訳で全体の傾向を、連続箇所（多い順）で「特に直したい場所」を把握できます。体言止めや問いかけを織り交ぜると、リズムが生まれて読みやすくなります。連続は同じ文末（例「ます。」）が 5 つ以上続いた箇所を、先頭の行番号で示します。

:::{.note}
文末の分類は形態素解析を使わない簡易判定です。「です・ます」「だ・である」は確実に判定しますが、「体言止め」は文末がひらがな以外（漢字・カタカナなど）で終わるものを名詞止めと見なす近似です。
:::

### よく使う言葉

本文に頻出する内容語（名詞・固有名詞・形容動詞・動詞・形容詞）を、品詞ラベルつきで上位 15 語まで挙げます。

```
🔤 よく使う言葉（内容語 上位15）

 1. 画像      名詞  306回
 2. ファイル  名詞  235回
 3. 生成      名詞  225回
 4. 設定      名詞  173回
 …
10. PDF       名詞  126回
```

同じ語ばかり使っていないか（言い換えの余地はないか）を確認できます。品詞ラベルは、製品名などの固有名詞（`Vivliostyle` など）や、`便利`・`重要` のような形容動詞も区別して表示します。助詞・助動詞などの機能語、`する`・`なる`・`ある`・`いる`・`できる` のような汎用動詞、記号は除外します。品詞の判定には形態素解析（MeCab）を使うため、MeCab を導入していない環境ではこのセクションは表示されません（導入状況は `vs doctor` で確認できます）。

:::{.note}
MeCab は単漢字（「章」など）を固有名詞と誤判定することがあるため、単漢字の固有名詞は集計から除いています（`Vivliostyle` のような正しい固有名詞は 2 文字以上なので残ります）。
:::

### 漢字レベル（ルビ候補）

本文の漢字を難易度レベルに分け、対象読者より上位の漢字＝ルビ（振り仮名）を振るべき候補を把握できます。

| レベル | 内容 |
|--------|------|
| 教育 (L0) | 小学校で習う漢字（学年別配当表） |
| 中学 (L1) | 中学で習う常用漢字 |
| 一般 (L2) | JIS第一水準の常用外漢字（人名・地名など） |
| 専門 (L3) | JIS第二水準の漢字（専門的・旧字体） |
| JIS外 (L4) | JIS X 0208 に無い漢字 |

```
🈂 漢字レベル（ルビ候補）

- レベル内訳: 教育 91% ／ 中学 9% ／ 一般(L2) 0% ／ 専門(L3) 0%
- 中学漢字（多い順）: 稿(77) 換(70) 索(61) …（ほか 164 字）
- 一般漢字(L2)（多い順）: 揃(10) 雛(9) 梱(8) …
- 専門漢字(L3)（多い順）: 敲(3) 俯(1) 閾(1) …
- 一般・専門漢字の出現箇所:
    閾 → 第33章 L384
    敲 → 第22章 L161, 第32章 L178, 第32章 L191
```

小学生向けに書く場合、中学以上の漢字にはルビが要ります。稀に出る一般・専門漢字は、出現の少ない順に**出現箇所（章・行）**を示すので、その場所を開いて `{漢字|よみ}` のルビ記法を書き足せます。

:::{.note}
`vs metrics` は「どの漢字がどのレベルで、どこに出るか」の統計に徹します。読みの生成（自動ルビ付与）は、まさに振り仮名を振りたい稀な漢字ほど形態素解析が誤読しやすく危険なため、将来の別コマンド `vs furigana`（半自動）に分離する予定です。
:::

:::{.section-lead}
metrics では、Web/技術書の黄金律に基づいた評価基準を使用しています。これらの基準は `config/book.yml` でカスタマイズ可能です。
:::

### 漢字比率

文章の「硬さ」を測定します。技術書では 25〜35% が理想的です。

| 範囲 | 評価 |
|------|------|
| 20% 未満 | 平易すぎる |
| 20〜25% | やや平易 |
| 25〜35% | 適切 |
| 35〜45% | やや難解 |
| 45% 超 | 難解 |

### 文字種構成

本文（コード除去後）を構成する文字の種別ごとの割合を示します。漢字・ひらがな・カタカナ・英字、および「その他」（数字・記号・句読点）に分けて表示します。漢字の割合は前述の「漢字比率」と同じ値です。

一般に、読みやすい和文はひらがなが多めで漢字が3割前後とされます（漢字3:ひらがな7 などの目安）。カタカナや英字が多い場合は、専門用語・外来語が多いことを示します。技術書では自然に高くなりますが、入門書で初学者向けに書くなら、カタカナ語の言い換えや補足のヒントになります。

### MATTR（語彙多様度）

表現の豊かさは MATTR（Moving-Average Type-Token Ratio／移動平均 TTR）で測定します。

素朴な TTR（異なり語数 ÷ 総語数）は、**文章が長いほど機械的に低くなる**という弱点があります。語彙（異なり語）は語数の平方根程度でしか増えないため、同じ書き手・同じ多様性でも、長い章ほど TTR が下がってしまい、章どうしを比較できません。

MATTR は、一定語数の「窓」を 1 語ずつずらしながら各窓の TTR を求め、その平均をとる指標です（Covington & McFall, 2010）。これにより**文書の長短に依存せず**、章をまたいだ語彙の多様性を公平に比較できます。窓幅は既定で 100 語です（後述のとおり設定で変更可能）。

| 範囲 | 評価 |
|------|------|
| 0.7 以上 | 非常に豊富 |
| 0.6〜0.7 | 豊富 |
| 0.5〜0.6 | 標準的 |
| 0.5 未満 | 単調 |

:::{.note}
窓付き移動平均という性質上、MATTR は素朴な TTR より高め・安定した値域になります。上表の境界は、実用書原稿の実測分布をもとにした目安です。
:::

### 読解難度

読解難度は、建石由佳・小野芳彦・山田尚勇による評価式（1988 年「日本文の読みやすさの評価式」）でスコア（RS）を算出します。文長や、漢字・ひらがな・カタカナ・アルファベットの「連（同じ文字種の連なり）」の平均長、句読点の比率といった、形態素解析に頼らず決定的に測れる量から導かれる、日本語リーダビリティー研究の代表的な指標です。

**RS は値が大きいほど読みやすい**ことを意味します。漢字の連なりが長いほど、また一文が長いほどスコアは下がります。

| RS スコア | 評価 | 対象読者 |
|-----------|------|----------|
| 60 以上 | Easy | 小中学生向け |
| 40〜60 | Standard | 一般的なビジネス・実用書 |
| 40 未満 | Professional | 専門家・技術者向け |

技術書はもともと漢字や専門用語の密度が高く、Standard〜Professional に収まるのが自然です。バンドの境界値は `config/book.yml` で調整できます（後述）。

## 設定のカスタマイズ

:::{.section-lead}
`config/book.yml` の `metrics` セクションで、評価基準をプロジェクトに合わせてカスタマイズできます。
:::

### プリセットの選択

4種類のプリセットから選択できます。

```yaml
metrics:
  use: standard  # compact / standard / commercial / author_custom
```

| プリセット | 想定ページ数 | 章の理想分量 |
|------------|--------------|--------------|
| compact | 20〜50ページ | 1,500〜3,000 文字 |
| standard | 100〜200ページ | 5,000〜10,000 文字 |
| commercial | 200ページ以上 | 8,000〜12,000 文字 |
| author_custom | カスタム | 自由に設定 |

### 除外章の設定

前書き、付録、後書きなどを分量警告から除外できます。

```yaml
metrics:
  exclude_chapters: [00, 90-98, 99]
```

### カスタム基準の設定

プリセットを上書きして独自の基準を設定することも可能です。

```yaml
metrics:
  use: author_custom
  author_custom:
    chapter:
      min: 2000
      ideal: [4000, 8000]
      max: 12000
    section:
      min: 500
      ideal: [1000, 2000]
      max: 3000
```

### 読解難度バンドの調整

読解難度のラベル（Easy / Standard / Professional）を分ける RS のしきい値も調整できます。`easy` 以上を Easy、`standard` 以上を Standard、それ未満を Professional と判定します（既定値はそれぞれ 60 と 40）。

```yaml
metrics:
  readability:
    easy: 60     # この値以上を Easy（やさしい）
    standard: 40 # この値以上を Standard（標準）
```

### MATTR の窓幅

語彙多様度（MATTR）の窓幅も変更できます。既定は 100 語です。先行研究では 50〜100 が標準で、これより小さいと推定が不安定に、大きくしても効果は頭打ちになります。

```yaml
metrics:
  mattr_window: 100 # MATTR の窓幅（語数）。50〜100 が目安
```

## ライブ表示とキャッシュ

:::{.section-lead}
metrics コマンドには、大規模プロジェクトでも快適に使えるライブ UI とキャッシュ機構が組み込まれています。
:::

### ストリーミング更新

`vs metrics` は 3 フェーズで進行します。

1. **サマリ即時出力** — 章の解析前に基礎統計と語彙・読解難度を表示。全章キャッシュが揃っていればキャッシュのみで算出されます。
2. **ライブプレースホルダー** — 章ごとの軽量スキャン結果（タイトル・概算文字数）をプレースホルダーとして描画し、解析完了次第リアルタイムで差し替えます。
3. **最終出力** — `--all`/`--warn`/章指定のフィルタを適用した後、確定した章リストを整形して表示。対象章が無い場合は UI が自動で「対象章がありません」を表示します。

ターミナルでは ANSI 制御を用いてグラフ部分だけを差し替えるため、大量のスクロールを起こさず進捗が把握できます。

### catalog.yml 連動

`config/catalog.yml` に記載された章のみが解析対象となります。除外された章（コメントアウトや未記載の章）は自動的にスキップされるため、必要な章だけを効率的に分析できます。

```yaml
# config/catalog.yml
CHAPTERS:
  - 01-intro
  - 02-basics
  # - 03-advanced  # コメントアウトされた章はスキップ
```

### キャッシュ機能（章単位の鮮度判定）

章ごとの解析結果は `.cache/metrics/{basename}.yml` に保存され、**章ファイル自身の最終更新日時 (mtime)** と比較して鮮度を判定します。`00-preface.md` など単一ファイルに依存した判定ではなくなったため、個別章の修正だけでキャッシュ差分更新が行われます。

キャッシュ内容は以下を含むため、再解析せずともメトリクスを完全に再構築できます。

- `basic_stats`: 文字数・行数・文/節統計
- `vocabulary_stats`: `tokens_map` (token => count)、MATTR、漢字文字数、総語長など
- `readability`: スコアとラベル、および全体集計用の特徴量（文・連・句読点の数）
- `sections`: 節ごとの文字数と警告

この情報は JSON/YAML 出力や集約サマリ計算でもそのまま再利用されます。

```bash
# キャッシュを無効にして実行
VIVLIO_METRICS_CACHE=0 vs metrics
```

環境変数 `VIVLIO_METRICS_CACHE=0` を設定すると、読み書きともに無効化され、常にフル解析が実施されます。

### 並列処理と再利用

章の解析は自動的に並列実行されます。CPUコア数に応じて並列度が自動決定され、大規模プロジェクトでも高速に処理されます。キャッシュ済み章はスキップされるため、変更の無い章はライブ表示が即座に最終表示へ切り替わります。

- デフォルト並列度: CPUコア数と4の小さい方
- 並列度の上書き: `VIVLIO_METRICS_CONCURRENCY` 環境変数

```bash
# 並列度を2に制限
VIVLIO_METRICS_CONCURRENCY=2 vs metrics
```

### 構造化出力と再計算不要のトークンマップ

`--json` / `--yaml` は以下の形式で出力します。

```json
{
  "stats": [
    { "path": "contents/01-intro.md", "chars": 1200, ... }
  ],
  "totals": {
    "chars": 2400,
    "avg_sentence_chars": 210.5,
    "readability": { "score": 42.1, "label": "Standard" },
    ...
  }
}
```

`totals` は Runner が各章の `tokens_map` / `basic_stats` / 読解難度の特徴量を合算した結果で、外部ツールが平均を再計算する必要はありません。読解難度（RS）は、章ごとの RS を単純平均するのではなく、各章の連や文の数といった特徴量を合算したうえで一度だけ算出します。語彙多様度（MATTR）は文書長に頑健な指標のため、全体値は各章 MATTR の語数加重平均で代表させます。いずれもキャッシュに保存済みのため、章を再解析せずに再構成できます。

## まとめ

:::{.section-lead}
metrics を活用することで、執筆中の原稿を客観的に評価し、読者にとって最適なバランスの技術書を作成できます。
:::

Vivlio Starter の metrics 機能により、以下のメリットが得られます。

- **客観的な評価**: 感覚に頼らず、数値で文章品質を把握
- **バランスの改善**: 章ごとの分量の偏りを可視化して調整
- **読みやすさの向上**: 語彙難度や読解難度を適切な範囲に維持
- **継続的な監視**: 執筆中に定期的にチェックして品質を維持

まずは `vs metrics` コマンドを実行して、現在の原稿の状態を確認してみましょう。
