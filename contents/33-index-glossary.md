# 索引・用語集機能

:::{.chapter-lead}
Vivlio Starter では、`vs index` コマンドを使用して、書籍の索引と用語集を効率的に作成できます。本章では、用語の登録から自動抽出、レビュー、索引・用語集ページ生成までの一連のワークフローを解説します。
:::

`vs index` コマンドは、原稿内の手動マークアップと自動抽出を組み合わせて、索引・用語集候補を統合管理します。TF-IDF スコアリングによる候補の優先順位付けと、Markdown ベースのレビューワークフローにより、効率的な索引・用語集作成を実現します。

## 索引・用語集作成の概要

:::{.section-lead}
索引・用語集作成は、用語の登録、候補の抽出、レビュー、ページ生成という4つのステップで構成されます。
:::

### 基本的なワークフロー

```bash
# 0. 原稿にタグを付与する
# contents/01-intro.md などを編集して [用語|読み] を追加

# 1. 索引・用語集候補の自動抽出とレビューファイル生成
vs index:auto

# 2. _index_glossary_review.md を編集して承認・棄却
# エディタで開いて [i]=索引, [g]=用語集, [ig]=両方, [r]=棄却

# 3. レビュー結果を適用
vs index:apply

# 4. 索引・用語集ページを含む本文 PDF 生成
vs build
```

### 関連ファイル

索引機能で使用・生成されるファイル：

| ファイル | 役割 |
|:---|:---|
| `config/index_glossary_terms.yml` | 承認済み用語の統合辞書（索引・用語集共通） |
| `config/index_glossary_rejected.yml` | 棄却済み用語のリスト |
| `_index_glossary_review.md` | レビュー用 Markdown ファイル |
| `_indexpage.html` | 生成された索引ページ |
| `_glossarypage.html` | 生成された用語集ページ |

## 手動マークアップ

:::{.section-lead}
原稿内で索引に載せたい用語を明示的にマークアップできます。手動マークアップは最も確実な索引登録方法です。
:::

### 基本的な記法

原稿内で `[用語|読み]` または `[用語]` の形式でマークアップします：

```markdown
[Ruby|るびー]は素晴らしいプログラミング言語です。

[JavaScript]を使ってウェブページに動きを付けます。
```

- `[用語|読み]` - 用語と読み（ひらがな）を指定
- `[用語]` - 用語のみ（読みは MeCab で自動推測）

:::{.note}
**読みの指定について**

日本語の用語は、索引で五十音順に並び替えるために読みが必要です。読みを省略した場合、MeCab（形態素解析エンジン）で自動推測されますが、固有名詞や専門用語は誤認識される場合があります。重要な用語は読みを明示することを推奨します。
:::

### 特殊文字のマークアップ

記号や HTML タグなども索引に登録できます：

```markdown
[!]は否定演算子です。
[&&]は論理積演算子です。
[||]は論理和演算子です。
[<h1>]は見出しタグです。
```

### マークアップの除外

以下の記法は索引として認識されません：

- `[リンクテキスト](URL)` - Markdown リンク
- `[^1]` - 脚注参照
- コードブロック内の `[...]` - コードフェンス内は無視

## 自動抽出とレビュー

:::{.section-lead}
`vs index:auto` コマンドは、原稿から索引候補を自動抽出し、レビュー用の Markdown ファイルを生成します。
:::

### 自動抽出の実行

```bash
vs index:auto
```

このコマンドは以下の処理を行います：

1. **手動マークアップの検出** - 原稿内の `[用語]` 記法を検出
2. **候補の自動抽出** - 定義パターン、専門用語、名詞連続を検出
3. **TF-IDF スコアリング** - 候補に重要度スコアを付与
4. **レビューファイル生成** - `_index_glossary_review.md` を生成

### スコアリングの仕組み

候補のスコアは以下の要素から算出されます：

| 要素 | 説明 | 重み |
|:---|:---|:---|
| TF（出現頻度） | 文書内での出現回数 | 1.0 |
| IDF（文書頻度の逆数） | 特定の章にのみ出現する用語を重視 | 5.0 |
| 定義パターン | 「〜とは」「〜について」などの定義文 | 30.0 |
| 専門用語パターン | カタカナ語、英語、略語 | 15.0 |
| 見出し近傍 | 見出しの近くに出現 | 20.0 |

### レビューファイルの構成

生成される `_index_glossary_review.md` は4つのセクションで構成されます：

```markdown
# 索引・用語集レビュー
※ フラグ: [i]=索引のみ、[g]=用語集のみ、[ig]=両方、[r]=棄却、[-i]=索引から除外、[-g]=用語集から除外
※ 読みの修正は ( ) 内を編集。用語集の説明文は空行の後にインデントして記述。

## 1. 登録済み用語の確認 (Terms: N語)
- [ig] `Today` **Ruby** (るびー) - [手動登録]
  - 01-intro: Rubyは素晴らしい...

  まつもとゆきひろが作成したプログラミング言語。

## 2. 推奨候補 (High Candidates: N語)
- [ ] `NEW!` **JavaScript** (じゃばすくりぷと) - スコア: 250.5
  - 02-basics: JavaScriptを使って...

## 3. 一般候補 (Low Candidates: N語)
- [ ] `NEW!` **変数** (へんすう) - スコア: 150.2
  - 03-variables: 変数とは...

## 4. 除外済みリスト (Rejected: N語)
※ 復帰させたいものは [i], [g], [ig] を入れると候補に戻ります。
- [ ] `Today` **フォー** (フォー) - スコア: 340.7
  - 00-preface: お問い合わせフォーム...
```

Section 1: Terms（確定済み 200語）
    ここは「既知」なので、著者は間違っているものがないか、流し読みするだけでOK。
Section 2: High-Score Candidates（厳選候補 50語）
    スコアが高いもの。「これは索引に入れるべきだよね」と同意しやすいものから並べる。
Section 3: Low-Score Candidates（保留候補 150語）
    「一応抽出したけれど、たぶん不要かも」というグループ。著者は余裕がある時だけ見れば良い。


:::{.note}
**新着ラベルについて**

執筆の参考となるよう、`NEW!` と `TODAY` が付いている候補は直近の抽出結果を示しています。

| ラベル | 説明 |
|:---|:---|
| `NEW!` | 直近３時間以内に抽出された索引用語 |
| `TODAY` | 今日抽出された索引用語 |
| なし | それ以外の索引用語 |
:::

### レビューの操作

| フラグ | 説明 |
|:---|:---|
| `[i]` | 索引のみに登録 |
| `[g]` | 用語集のみに登録（説明文を記述可能） |
| `[ig]` | 索引と用語集の両方に登録 |
| `[r]` | 両方から棄却（除外リストに追加） |
| `[-i]` | 索引からのみ除外 |
| `[-g]` | 用語集からのみ除外 |
| `[ ]` | 保留（次回のレビューまで保持） |

:::{.note}
**読みの修正**

読みを修正したい場合は、括弧内を直接編集します：

```markdown
- [ig] **Ruby** (るびー)  →  - [ig] **Ruby** (ルビー)
```
:::

:::{.note}
**用語集の説明文**

用語集に登録する場合、出現箇所のリストの後に空行を入れ、インデントして説明文を記述します：

```markdown
- [g] `NEW!` **非同期処理** (ひどうきしょり) - スコア: 295.6
  - 00-preface: 非同期処理はJavaScriptを日々利用して
  - 01-life: プログラミングでは非同期処理を行ないます

  処理の完了を待たずに*次のタスクを実行*する方式。
  `JavaScript`の`Promise`などで利用されます。
```

空行がないと出現箇所と説明文の境界が判別できません。
:::

### レビュー結果の適用

編集が完了したら、結果を適用します：

```bash
vs index:apply
```

このコマンドは以下の処理を行います：

- `[i]` `[ig]` マークされた候補を `config/index_glossary_terms.yml` に索引用語として追加
- `[g]` `[ig]` マークされた候補を `config/index_glossary_terms.yml` に用語集用語として追加
- `[-i]` マークされた用語から索引フラグを除去
- `[-g]` マークされた用語から用語集フラグを除去
- `[r]` `[-ig]` マークされた候補を `config/index_glossary_rejected.yml` に追加
- 読みの変更を反映

## 索引データの生成

:::{.section-lead}
`vs build` コマンドは、原稿をスキャンして索引用語にタグを付与し、索引データを含む本文 PDF を生成します（内部の仕組みに興味がある人向けの情報です）。
:::

### マッチングの実行

```bash
vs build
```

ビルド時、内部では以下の処理が行われます：

1. `config/index_glossary_terms.yml` の用語（`flags` に `i` を含むもの）を原稿から検索
2. マッチした箇所に HTML タグを付与
3. 索引・用語集データを生成

### 生成されるタグ

原稿内の用語は以下のタグに変換されます：

```html
<!-- 初出（定義） -->
<dfn id="idx-ruby-1" class="index-term" data-yomi="るびー">Ruby</dfn>

<!-- 2回目以降 -->
<span id="idx-ruby-2" class="index-term" data-yomi="るびー">Ruby</span>
```

- **初出**: `<dfn>` タグ（定義を示す）
- **2回目以降**: `<span>` タグ

## 索引ページの生成

:::{.section-lead}
`vs build` コマンドは、索引データから索引ページを生成します。
:::

### 索引ページの生成

```bash
vs build
```

生成される `_indexpage.html` は以下の特徴を持ちます：

- **五十音順ソート** - 読みに基づいて「あ行」「か行」などでグループ化
- **英数字対応** - `A-Z`、記号、数字も適切にグループ化
- **CSS target-counter 対応** - ページ番号は CSS で自動挿入
- **重複排除** - 同一ページ内の複数出現は1つにまとめる

## 用語集ページの生成

:::{.section-lead}
用語集ページは、登録された用語とその説明文、本文へのバックリンクを一覧表示します。
:::

### 用語集ページの特徴

生成される `_glossarypage.html` は以下の特徴を持ちます：

- **五十音順ソート** - 「あ行」「か行」などでグループ化（カタカナ語も対応）
- **Markdown 対応** - 説明文内の `##` 記法や `*` リストを HTML に変換
- **バックリンク** - 本文での出現箇所へのリンク（ページ番号表示）
- **重複排除・昇順ソート** - バックリンクはページ番号の昇順で表示、同一ページは1つにまとめる

### バックリンクの表示

用語集ページの各用語には、本文での出現ページが表示されます：

```
→ p.1, 5, 12, 45
```

- ページ番号のみ表示（章名は省略）
- 重複するページは自動的に排除
- ページ番号の昇順でソート
- クリックで本文の該当箇所へジャンプ

### 索引ページの構造

```html
<section class="index">
  <h1>索引</h1>
  <div class="index-section" data-initial="あ">
    <h2>あ</h2>
    <dl class="index-list">
      <dt>アルゴリズム</dt>
      <dd><a href="03-basics.html#idx-algo-1"></a></dd>
    </dl>
  </div>
  <!-- 他の行も同様 -->
</section>
```

## 設定のカスタマイズ

:::{.section-lead}
`book.yml` の `index` セクションで、索引機能の動作をカスタマイズできます。
:::

### 索引設定

```yaml
index:
  # 基本設定
  enabled: true                  # 索引機能を有効化
  auto_discovery: true           # 自動抽出を有効化（falseで手動索引用語のみ）

  # スコア閾値
  auto_approve_threshold: 300    # この値以上は自動承認
  review_threshold: 150          # この値以上がレビュー対象
  high_candidate_ratio: 0.25     # 推奨候補の割合（上位25%）
  
  # 表示設定
  context_width: 40              # 文脈の表示幅（文字数）
  timezone: 'Asia/Tokyo'         # タイムゾーン（ラベル判定用）
```

### 用語集設定

```yaml
glossary:
  # 基本設定
  enabled: true                  # 用語集機能を有効化
  title: '用語集'                # 用語集ページのタイトル

  # 説明文設定
  require_definition: false      # 説明文を必須にするか
  max_definition_length: 200     # 説明文の最大文字数
```

:::{.note}
**用語集リンクについて**

本文中の用語集登録語には、自動的に上付きの `†`（ダガー）記号が付与されます。この記号はアクセント色（`book.yml` の `theme.color` 設定に連動）で表示され、クリックすると用語集ページの該当項目へジャンプします。

例: コンピュータ<sup>†</sup> → クリックで用語集へ
:::

### 自動抽出機能

`Vivlio Starter`は、索引の候補となる用語の自動抽出機能を実装しています。
技術書の索引の目安は、一般的に**1ページあたり1〜2語**程度と言われています。200ページの本であれば、最終的な索引語が 200〜300語 程度に収まっていると、読者にとって非常に使い勝手の良い、引きやすい索引になります。

| 設定 | デフォルト | 説明 |
|:---|:---|:---|
| `auto_approve_threshold` | 300 | この値以上のスコアを持つ候補は自動的に承認 |
| `review_threshold` | 150 | この値以上のスコアを持つ候補がレビュー対象 |
| `high_candidate_ratio` | 0.25 | レビュー候補のうち、推奨候補として表示する割合 |

| 目安語数 | 状態 | 説明 |
|:---|:---|:---|
| 400語 | 詳細な索引 | かなり細かいキーワードまで拾っている状態。著者のレビュー負担は大きいが辞書的価値が高い。 |
| 200〜300語（推奨） | 標準的な索引 | 重要な概念や専門用語がしっかり網羅され、技術書として最も標準的。 |
| 100語以下 | 情報不足の索引 | 主要な章のテーマしか拾えておらず、索引として物足りない印象。 |

閾値を設定することで、自動承認とレビュー対象を調整できます。執筆中の書籍に合わせて最適な値を設定してください。

## 実用的なワークフロー

:::{.section-lead}
実際の書籍制作における索引作成の典型的なワークフローを紹介します。
:::

### 索引作成

```bash
# 1. 原稿に重要な用語をマークアップ
# contents/01-intro.md などを編集して [用語|読み] を追加

# 2. 自動抽出とレビューファイル生成
vs index:auto

# 3. _index_glossary_review.md を開いてレビュー
# [i] で索引登録、[g] で用語集登録、[ig] で両方、[r] で棄却

vs index:apply

# 5. （原稿完成後、索引ページも含めて）PDFをビルドして確認
vs build
```

## トラブルシューティング

:::{.section-lead}
索引機能の実行時によくある問題とその解決方法を紹介します。
:::

### MeCab が利用できない

**症状**: 読みの自動推測が動作しない

**解決方法**:

vivlio-starter では、以下のコマンドで MeCab と natto を自動的にインストールできます:

```bash
vs doctor --fix
```

または手動でインストールすることもできます:

```bash
brew install mecab mecab-ipadic

# Ruby gem
gem install natto
```

### 索引が五十音順にならない

**症状**: 索引ページで用語が正しくソートされない

**原因**: 読みが正しく設定されていない

**解決方法**:
1. `_index_glossary_review.md` を確認
2. 読みが誤っている用語の ( ) 内を修正
3. `vs index:apply` を実行し修正を適用
4. `vs build` を実行してPDFを確認

### コードブロック内の用語が抽出される

**症状**: コード内の変数名などが候補に含まれる

**原因**: コードブロックの記法が正しくない

**解決方法**:
- コードブロックは ` ``` ` で正しく囲む
- インラインコードは `` ` `` で囲む

### 用語が索引に載らない

**症状**: 原稿中に頻出する用語なのに、索引ページに表示されない

**原因**: 自動索引機能が有効になっていない

索引システムは以下の2つの方法で用語をタグ付けします：

1. **手動マークアップ**: 原稿内で `[用語]` や `[用語|読み]` と明示的に記述
2. **自動タグ付け**: `config/index_glossary_terms.yml` に登録された用語を自動検出

`vs index:auto` や `vs index:apply` を実行していない場合、`config/index_glossary_terms.yml` が生成されず、手動マークアップした用語のみが索引に載ります。

**解決方法**:

```bash
# 1. 自動抽出を実行してレビューファイルを生成
vs index:auto

# 2. _index_glossary_review.md を開いて [i]/[g]/[ig] で承認
# （エディタで編集）

# 3. 承認結果を適用
vs index:apply

# 4. 再ビルド
vs build
```

:::{.column}
**ヒント**: `vs build` 実行時に `config/index_glossary_terms.yml` が見つからない場合、警告メッセージが表示されます。自動索引機能を活用するには、まず `vs index:auto` を実行してください。
:::

## 既知の制約（用語集のページ番号表示）

:::{.section-lead}
Vivliostyle における `target-counter(attr(href url), page)` は、リンク先 HTML を個別にレンダリングした際の「章ローカル」なページ番号しか取得できません。
:::

- 用語集のバックリンクに表示される `p.1, 5, 12...` などの数字は、章ごとにリセットされるローカルページ番号です。
- PDF ビューワ上でリンクをホバーすると表示される「◯ページに移動」は、最終出力PDFの絶対ページ番号です。
- 現状の CSS / Vivliostyle だけでは両者を一致させる手段がないため、この差異は制御不能な既知の制約として扱います。

## まとめ

:::{.section-lead}
`vs index:auto` コマンドを活用することで、索引作成作業を大幅に効率化できます。
:::

本章で学んだ内容：

- ✅ 手動マークアップによる索引用語の登録 `[用語|読み]`
- ✅ `vs index:auto` による自動抽出とスコアリング
- ✅ `_index_glossary_review.md` でのレビューワークフロー
- ✅ `vs index:apply` による承認・棄却の適用
- ✅ `book.yml` でのカスタマイズ

索引は書籍の利便性を大きく向上させる重要な要素です。`vs index:auto` コマンドの自動抽出機能とレビューワークフローを活用することで、漏れのない索引を効率的に作成できます。

:::{.column}
**次のステップ**

索引ページを生成したら、`vs build` コマンドで本文 PDF に統合されます。索引ページのスタイルは `stylesheets/index.css` でカスタマイズできます。
:::
