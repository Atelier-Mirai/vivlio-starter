# Vivlio Starter: Text Metrics Specification

## 1. 目的

執筆中の原稿を定量的に分析し、著者が内容の「難易度」「表現の多様性」「構成のバランス」を客観的に把握・改善するための指標を定義する。

## 2. 集計指標

### 2.1 基本統計量 (Basic Stats)

最も基礎となる分量の指標を以下のカテゴリで集計する。

#### 📊 文章統計 — 基本情報
- **文字数**: 12,345 文字（改行除く）
- **行数**: 123 行

#### 📊 文章統計 — 文構造
- **文の数**: 45 文（平均 274 文字/文）
- **節の数**: 134 節（平均 92 文字/節）
- **読点数**: 89 個

### 2.2 語彙難度 (Lexical Difficulty)

文章の「硬さ」や「専門性」を測定する。

- **漢字比率 (Kanji Ratio)**  
  全文字数に対する漢字（`\p{Han}`）の割合。25〜35% が実用書として理想的。
- **平均語長 (Average Word Length)**  
  分子: 全文字数、分母: MeCab による分かち書き後のトークン総数。長い専門用語が増えると値が上昇する。

### 2.3 語彙多様度 (Vocabulary Diversity)

表現の豊かさや単調さを測定する。

- **TTR (Type-Token Ratio)**  
  異なり語数 ÷ 総語数。MeCab の基本形 (`base_form`) を用いて活用の違いを正規化する。
- **TTR の評価基準**
  - 0.7 以上: 非常に豊富
  - 0.5〜0.7: 豊富
  - 0.3〜0.5: 標準的
  - 0.3 未満: 単調

### 2.4 読解難度スコア (Readability Score)

日本語版 Flesch/Kincaid を応用した読みやすさ指標。

- **算出式（プロトタイプ）**

  ```
  score = (平均文長 × 0.5) + (漢字比率 × 0.5) + 難読語カウント
  ```

- **評価ラベル**
  - **Easy**: 小中学生向け
  - **Standard**: 一般的なビジネス・実用書
  - **Professional**: 専門家・技術者向け

### 2.5 章・セクション単位の可視化 (Visual Balance)

構成の歪みを可視化する。

- **算出単位**: Markdown の H1 または H2 単位での文字数
- **表示形式**: 最大分量を 100% とした相対的な水平棒グラフをターミナルに描画

## 3. 実装要件 (Ruby 4.0 Standard)

### 3.1 解析エンジン
- **MeCab**: 索引用語抽出で用いる辞書設定を流用し、形態素解析を実施。
- **Pattern Matching**: `case ... in` を活用し、品詞（助詞・名詞・動詞）ごとの比率計算を最適化。

### 3.2 実行タイミング
- `vs metrics` コマンドを単独実行し、詳細レポートとして提供する。

## 4. Metrics の既定値

Web/技術書の黄金律に沿った既定値を `config/book.yml` の `metrics` セクションで管理し、必要に応じて上書きできるようにする。

| 状態           | 文字数の目安 |
|----------------|--------------|
| 標準（目標）   | 1,000〜1,500 文字 |
| 警告（やや長い） | 2,000 文字超      |
| 警告（加筆検討） | 800 文字未満      |

※ その他の基準値も、config/book.yml の metrics セクションで上書き可能。

## 3.5 解析対象・実行フロー・高速化

### catalog.yml に基づく章のフィルタリング

- `metrics` は `config/catalog.yml` の PREFACE/CHAPTERS/APPENDICES/POSTFACE に記載された章のみを解析対象とする。
- カタログに存在しない Markdown（下書き、退避用など）は一覧に含めない。
- `vs metrics 11-13` のように明示的に章を指定した場合は、その指定を優先し、カタログ外でも実行する。

### 出力フロー

1. コマンド実行直後に、対象章を軽量スキャン（`File.read` ベースで文字数を取得）し、章順にプレースホルダーのバー表示を行う。この時点では警告や節情報は表示せず、「第00章 … 1,857 文字」のような簡易行を順次出力し、進捗表示とする。
2. 軽量スキャンと並行して章別の詳細解析（Analyzer / ChapterParser）を行い、各章の処理が完了した時点で該当行を最終版（バー＋警告＋節表示）に差し替える。`--all`/`--warn`/章指定の各モードでも同じ挙動とする。
3. サマリ（基本統計/文構造/詳細分析）は従来通り全章を対象に算出し、章別バーのプレースホルダー出力と干渉しないタイミングで表示する。
4. JSON/YAML 出力では従来通り一括出力とし、上記の逐次表示はテキストモード限定とする。

### キャッシュ

- `~/.cache` ではなく、プロジェクト配下の `.cache/metrics/` を利用する。
- 章ごとに `{basename}.yml` 形式で解析結果を保存する。YAML で保存しておくことで、人間が直接確認・調整しやすい。
  - 保存項目の例:
    - ベース情報: `path`, `title`, `chapter_num`, `chars`, `warning`
    - 節情報: 各 `sections` の `title`, `chars`, `warning`
    - 基本統計: `lines`, `chars_without_newline`, `sentences`, `avg_sentence_chars`, `commas`, `clauses`, `avg_clause_chars`
    - 語彙統計: 以下の値を保持し、章キャッシュだけで全体値を再計算できるようにする
      - 漢字比率算出用: `kanji_char_count`, `total_char_count`
      - 平均語長算出用: `total_word_length`, `total_words`
      - TTR 算出用: `tokens_map`（`{"コンピュータ": 5, "HTML": 3, ...}` のような語→出現回数の Hash）
    - 読解難度: スコアとラベル（Easy/Standard/Professional 等）
    - その他 Analyzer/WarningChecker が再計算なしで再現できる付帯情報
- TTR（Type-Token Ratio）は章キャッシュの `tokens_map` をマージし、ユニーク語種数＝マージ後のキー数、総語数＝値の合計として算出する。章キャッシュのみでユニーク語を重複カウントしない。
- 各章の Markdown ファイルと `.cache/metrics/{basename}.yml` の `mtime` を比較し、ソースが新しければ即座に再解析する（1 章ごとに鮮度判定する）。
- `VIVLIO_METRICS_CACHE=0` などの環境変数で無効化できるようにしておく。

### 並列処理

- 章解析は内部スレッドプールで並列実行する。
- 並列数は build コマンドと同様に `Etc.nprocessors` と上限値（既定 4）の小さい方を採用し、自動で決定する。
- 上級ユーザー向けに `VIVLIO_METRICS_CONCURRENCY` 環境変数を用意し、並列数を上書きできるようにする。
- 並列処理で得られた結果はキャッシュに書き込み、以後の実行で再利用する。

## 4. コマンド例
```zsh
# 基本: 章のみ表示（概要）
vs metrics

# 特定の章: 節まで表示
vs metrics 2              # 第2章のみ
vs metrics 1,3,5          # 第1、3、5章
vs metrics 1-3            # 第1章から第3章まで
vs metrics 1-3,5,8-10     # 範囲と個別指定の組み合わせ

# すべての章: 節まで表示
vs metrics --all

# 警告がある章のみ: 節まで表示
vs metrics --warn
```

※ 0, 1, ..., 9 は 00, 01, ..., 09 と見做す
※ vs metrics --detail=2のように、detailオプションを検討していたが、vs metrics 2 のように簡潔なコマンドとした。CLI 仕様として --detail は廃止し、位置引数＋--all/--warn を使うこととする。
※ 既存のTablePrinter等で出力を行なっているが、現在の仕様は全て廃止し、この仕様書に沿って実装を行なう。
