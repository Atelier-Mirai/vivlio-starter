# print_pdf フチなし（full_bleed）設計メモ — 写真集・爪見出しへ向けた事前整理

## 0. 前提用語

- **塗り足し（bleed）**: 仕上がり線（トリム）の外側に伸ばしておく余分。既定 3mm。
  裁断が多少ズレても、紙の端まで絵が続くようにするための保険。
- **フチなし要素（full-bleed element）**: 紙の端まで達するデザイン要素。
  例: 全面写真の扉絵・節絵、**爪見出し（つめ見出し／サムインデックス）**、背景ベタ。
- **版面**: 本文の文字組み・余白のレイアウト。トリムサイズ基準で設計される。

---

## 1. `full_bleed: true` / `false` の本質的な違い

両者とも**最終入稿 PDF のジオメトリは同じ**（MediaBox を bleed+crop 分だけ拡張し、トンボを付ける）。
違うのは **トリム端の外 3mm 帯（塗り足し帯）に、実際に中身があるか／無いか** だけ。

| | `false`（導出・既定） | `true`（従来の個別レンダ） |
|---|---|---|
| 入稿用本文の作り方 | 閲覧用 PDF を qpdf で座標変換して導出 | vivliostyle を `--bleed 3mm` 付きで再レンダ |
| 塗り足し帯 3mm の中身 | **白**（元 PDF に情報が無い） | **絵が実際に伸びている** |
| フチなし要素のある本 | ✗ 白フチ裁ち落とし事故 | ○ 正しく裁ち落とせる |
| 速度 | 速い（~35s） | 遅い（192.5s・再レンダ3回・flaky あり） |

### なぜ導出（false）だと塗り足しを埋められないか

導出の出発点である閲覧用 PDF（`_sections.pdf` 等）は、**トリムサイズちょうどで生成済み**で、
塗り足し帯そのものが存在しない。導出処理がやるのは「内容を (bleed+crop) だけ内側へ平行移動して
MediaBox を広げる」だけ（本仕様 §3.2）。**新しくできた 3mm 帯を埋める情報がどこにも無いので白**になる。
→ これが本仕様 §2.6「閲覧用 PDF はトリムで裁たれており塗り足しを復元できない」の実体。

## 2. 「拡大」ではない — よくある誤解の整理

`true`/`false` のどちらでも、**版面（トリムサイズのレイアウト）は原寸のまま拡大・縮小しない**。
本仕様 §1.2 の実測がそれを示す:

- 入稿用の **TrimBox 実寸 = 595.28 × 841.92pt = ぴったり A4**（`640.63 − 45.35 = 595.28`）。
- 広がっているのは **MediaBox（用紙そのもの）**。片側 **16mm = 塗り足し3mm + トンボ代13mm**
  （`CROP_MARK_OFFSET_MM = 13.0`）。

やっていることは「拡大」ではなく **「A4 の版面を原寸で用紙中央に置き、外周を広げてトンボを付ける」**。
この用紙拡張は `true`/`false` 両方で同じように起きる。

---

## 3. 塗り足し帯の「絵」はどこから来るか

要素の種類で 2 つに分かれる。

### ケース1: CSS で描かれる要素（爪見出し・背景ベタ・罫）

**素材の準備は不要**。CSS が「ページの端＝塗り足し帯まで塗れ」と指示していれば、
vivliostyle がレンダリング時に**その場で塗り足し帯まで矩形を描き込む**。ベクトルなので伸ばしても劣化しない。
→ 爪見出しはこのケース。CSS で塗り足し端まで色を伸ばせる。

### ケース2: ラスタ写真（例: 青空の全面写真）

**魔法は無い**。塗り足し帯を埋めるには著者が次のどちらかを用意する:

1. **最初から塗り足し込みのサイズで写真を用意**（DTP の標準作法）。
   仕上がり A4 なら「A4 + 周囲 3mm」を覆う原稿を作り、版面の外 3mm まで届くよう配置。
2. **少し拡大して塗り足し帯までオーバーさせる**（`object-fit: cover` 等）。
   トリムぴったりの写真しか無い場合の逃げ。全体がわずかに拡大＝周囲がトリミングされる。

### これが `true` が「再レンダ経路」を要する理由

どちらのケースも、**塗り足し帯の中身が確定するのはレンダリングの瞬間**（CSS がその場で描く／
著者の“塗り足し込み元画像”を CSS が塗り足し端まで敷く）。`true` は vivliostyle を塗り足し帯まで
含めて走らせるのでこの解決が行われる。`false` の出発点はトリム後 PDF で、元画像も CSS も失われている。

### 表紙（カバー）についての注意

**表紙は print_pdf に結合されず別入稿**（本仕様 §0・§1.1 手順1）。`full_bleed` の判定は表紙に効かない。
表紙写真は設定に関係なく著者が「塗り足し込みサイズ」で用意する責任があり、まさにケース2 そのもの。
`full_bleed` が実際に効くのは **本文ページ側のフチなし要素（全面扉絵・節絵、爪見出し）**。

## 4. 速度の内訳（target 別）

`false` が速いのは、**閲覧用 PDF のレンダリング成果物を再利用**するから（再レンダ 0 回）。

| ターゲット | 従来 | 導出後（本仕様） | ②(dedup高速化)併用時 |
|---|---|---|---|
| `pdf` ＋ `print_pdf` | pdf ＋ 192.5s | **pdf ＋ 約35s** | 同上 |
| `print_pdf` のみ | レンダ3回＋dedup | **ほぼ同等（相殺）** | 約70s 短縮 |

- **`pdf` ＋ `print_pdf`**: 閲覧用レンダは pdf ターゲットのためどのみち走るので、
  print_pdf はその成果物を qpdf 変換するだけ。実質「pdf ビルド時間 ＋ 約35s」。最大の勝ち筋。
- **`print_pdf` のみ**: `derive_print` が真だと `!t.pdf` でも `build overall pdf`〜
  `build front pages and tail` が走る（本仕様 §2.5）。「入稿用3回レンダ」の代わりに
  「閲覧用1回レンダ＋dedup＋変換」をやるので時間は**相殺**。導出化“単体”では速くならず、
  ②（dedup 高速化）併用で preview 分（約70s）が浮く。
- 約35s の内訳: qpdf 結合7s ＋ 変換13s ＋ outline15s。

### 速度以外の恒久メリット（target に依らず効く・本仕様 §6 副次効果）

- 入稿用と閲覧用の本文が**同一レンダリング由来**になり、ページずれ・内容差が構造的に消える
- 入稿用レンダの **flaky（body-guard リトライ）消滅**
- Standard モードのノンブル工程による **/Dests 破壊（潜在バグ）解消**

## 5. 当面の運用方針: `full_bleed` は「隠し設定」として実装

現時点でフチなし要素を持つデザインが無いため、常時 `false` 運用。ただし
**バタで定数ハードコードにはせず、CONFIG を読む形（既定 false）を残し、雛形と docs には出さない**。

### 実装の切り分け

- **やる**: 本仕様 §2.5 の判定 `derive_print = t.print_pdf && !truthy?(CONFIG.output.print_pdf.full_bleed)`
  をそのまま実装。CONFIG 既定値に `full_bleed: nil`（＝false 扱い）を追加（`common.rb:207` 付近）。
- **やらない（爪/写真集デザイン着手まで保留）**: 雛形 `config/book.yml` への行追加、原稿 41-book-yml 章への記述。

### なぜ「定数ハードコード」より良いか

1. **従来レンダ経路はどのみち残す必要がある**。本仕様 §3.6 に「`/Rotate ≠ 0` 検出時は導出を中止し
   従来フローへフォールバック」という安全弁があり、Phase 0 も従来フローを恒久的に残す前提。
   削れるのは book.yml の行と docs だけ。
2. **config 読み取りと定数ハードコードは同じ一行のコスト**。将来の差分が小さい方を選ぶ。
   config 読みにしておけば、将来は**雛形に1行足して docs を書くだけ**（コード・テスト無変更）。
3. **テスト資産と escape hatch を失わない**。§5.1 の「`full_bleed: true` で従来フロー」テストが
   config キーを立てるだけで書ける。先進ユーザーもコード改変なしに `true` で回避できる（degrade-safe）。

| | 見える挙動 | book.yml 行 | docs | 将来 爪/写真集が来たとき |
|---|---|---|---|---|
| 定数ハードコード | 常時 false | なし | なし | 判定行＋config＋docs を追加 |
| **config読む・非公開（採用）** | 常時 false | なし | なし | **雛形1行＋docs だけ** |

## 6. 将来案: フチなしの「拡大方式」（第3のモード候補）

`true` を実装するとき、192.5s の再レンダ経路を作る前に**まず検討する価値のある高速案**。
中身は §3 ケース2-②「少し拡大」を、写真1枚ではなく**ページ全体へ一括適用**するもの。

### 仕組み — 導出経路と同一機構、変換行列の1項だけ違う

- 導出（`false`）: `q 1 0 0 1 m m cm`（**平行移動だけ** → 塗り足し帯は白）
- 拡大方式: `q s 0 0 s tx ty cm`（**中心から s 倍に拡大** → 内容が塗り足し帯まではみ出す）

閲覧用 PDF を再利用し、qpdf `--update-from-json` でアフィン変換するだけ。**再レンダ 0 回**なので
**~35s**（qpdf結合7s ＋ 変換13s ＋ outline15s）。192.5s ではない。

### 拡大率は判型に反比例する（要注意）

塗り足しを全周へ届かせる拡大率は短辺 W と塗り足し b で決まる:

```
s = (W + 2b) / W        （アスペクト維持なら短辺基準）
```

b=3mm 固定でも、判型が小さいほど拡大率が跳ね上がる:

| 判型 | 短辺 | 拡大率 s |
|---|---:|---:|
| A4 | 210mm | **+2.86%** |
| A5 | 148mm | +4.05% |
| B6 | 128mm | +4.69% |
| 文庫(A6) | 105mm | **+5.71%** |

ここでの拡大は用紙拡張ではなく**版面ごと拡大**なので次が一律に効く:

- 本文級数が大きくなる（例: 文庫 8.5pt 本文 → 実質 9.0pt 相当）
- 設計した天地小口の余白が同率で詰まる（→ 予め余白を広めに組んで吸収する必要）
- 図版が同率で拡大（ノンブルは後載せなので無傷）
- 全周が s−1 分トリム外へ押し出され**四辺が少し裁ち落とされる**

### もう一つのトレードオフ: 全ページ一律

拡大方式は**フチなし要素の無い通常ページまで一律に拡大**する。全ページ共通の爪見出しなら問題ないが、
「数ページだけ全面扉絵、他は普通の本文」という**混在本**では本文ページまで太る。
ページごとに拡大有無を変えると版面が不揃いになるため、実質**本1冊単位で all-or-nothing**。

### 3モード比較

| モード | 本文の忠実度 | 速度 | フチなし | 向く本 |
|---|---|---|---|---|
| `false` 導出（現行） | 原寸 | ~35s | ✗ 白フチ事故 | フチなし要素の無い通常の本 |
| **拡大方式（将来案）** | 一律 +2.9〜5.7% | ~35s | ○ | A4・A5 中心／混在せず全面フチなし基調のビジュアル本 |
| `true` 再レンダ | 原寸（CSS で厳密） | 192.5s | ○ | 文庫・混在本・厳密な級数指定の本 |

### 推奨する将来判断

- 写真集など**A4/A5・全面フチなし基調**の本 → **拡大方式**で再レンダ経路そのものを不要化できる可能性。
  +3〜4% を織り込んで版面を組むのが前提。
- **文庫・混在本・厳密な級数指定**の本 → 歪みが効くので `true` 再レンダを残す。
- 本仕様 §2.6 の脇に「`full_bleed: true` の実装候補は (a) 再レンダ / (b) 拡大方式。
  判型と混在有無で選択」とメモを残しておくと将来の判断が速い。
