# system_pages_cache_spec.md

## 背景
`_titlepage.md`, `_legalpage.md`, `_colophon.md` はシステム生成ページであり、利用者が執筆する原稿とは切り離して扱いたい。`contents/` に置きっぱなしだと、原稿と混在して誤編集や不要な Git 差分を生む。

## 現状の課題
- `contents/` にシステムページが残るため、原稿ディレクトリの役割が不明瞭。
- システムページを誤って編集・削除してもすぐには気づきにくい。
- クリーンなプロジェクト構成を保つために、利用者が余計なファイルに気を配る必要がある。

## 新仕様
1. **生成場所**: システムページは `.cache/vs/` 直下にのみ生成・保持する。パスは `File.join('.cache', 'vs', filename)` をその場で計算する。
2. **自動生成**: `pre_process`（もしくは Step 3 の開始直前）で `_titlepage.md` `_legalpage.md` `_colophon.md` の存在を直接 `File.exist?` で確認し、不足していれば既存テンプレートロジックで `.cache/vs/` に書き出す。
3. **ビルド入力**: 章リストを構築する際、`contents/` のユーザー原稿に加えて `.cache/vs/` のシステムページを所定の順序で追加する。`contents/` 側に同名ファイルがあっても参照しない。
4. **キャッシュ扱い**: `.cache/vs/` が削除されていても、次回ビルド開始時に同じ手順で再生成するだけでよい。生成コストは小さいためパフォーマンスへの配慮は不要。
5. **ユーザー操作**: `book.yml` の設定に応じてビルド時に自動生成・更新される（現行仕様と同じく、利用者が直接操作する導線は設けない）。

## 実装方針
1. `pre_process` 入口で `.cache/vs/` ディレクトリを `FileUtils.mkdir_p` で確保する。
2. `['_titlepage.md', '_legalpage.md', '_colophon.md']` を順にチェックし、存在しない場合のみテンプレート生成メソッドを呼んで `.cache/vs/` へ書き込む。
3. 章リストを組み立てる際は、`.cache/vs/` のファイルをタイトルページ→リーガル→コロフォンの順で先頭/末尾に差し込む。`contents/` 側のコピーが残っていても無視するため、特別な後方互換処理は不要。
4. 追加のヘルパークラスは作らず、既存処理内で条件分岐とファイルコピーを完結させる。簡潔な if 文で把握できる実装とし、ログは「生成した/既存を利用した」程度に留める。

## 移行
- `contents/` から `_titlepage.md` などはすでに削除済みという前提で即時移行する。`contents/` 内に古いファイルが残っていても参照しないため、互換レイヤーは持たない。
