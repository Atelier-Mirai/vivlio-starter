# vivlio-starter: 索引生成機能の仕様案とアルゴリズム

## 0. 前提環境

- **Ruby 4.0.0+** を必須とする
- Ruby 4.0.0 で Set が組み込み化され、`require "set"` が不要に
- Set リテラル構文 `Set[]` が利用可能
- 内部的に最適化された Set を活用し、高速な用語管理を実現

## 1. 基本コンセプト
Vivliostyle（CSS組版）では、ページ番号はブラウザでのレンダリング時に確定するため、Ruby（ビルド時）には不明です。そのため、**「Rubyでリンク構造を作り、CSSでページ番号を流し込む」**というアプローチをとります。

## 2. 実装アルゴリズム（全体像）

### Step 1: 索引語のマークアップ（ソース）
著者が索引にしたい語句をMarkdown上で指定します。

**手動指定の2つの形式:**

1. **完全形（読み付き）**: `[用語|よみ]`
   - 例: `[レスポンシブデザイン|れすぽんしぶでざいん]`
   - 例: `[引数|ひきすう]` （読み間違いしやすい語句）
   - `{漢字|ふりがな}` に倣った記法で、著者にとって違和感なく使える
   - 脚注 `[^1]` と同様に `[]` で囲む形式

2. **簡略形（読み省略）**: `[用語]`
   - 例: `[基本情報技術者]` （日本人なら読める語句）
   - 例: `[レスポンシブデザイン]`
   - MeCab で読みを自動推測
   - `vs index:match` で確認・修正可能

- **自動抽出:** 後述するアルゴリズムにより、見出しや強調箇所から自動抽出。

### Step 2: HTML変換（Ruby 4.0.0 処理）
ビルド時に、対象語句をID付きのアンカー要素に変換します。

#### 2.1 手動記法の変換
- **変換前:** `[レスポンシブデザイン|れすぽんしぶでざいん]とは...`
- **変換後（初出）:** `<dfn id="idx-..." class="index-term" data-yomi="れすぽんしぶでざいん">レスポンシブデザイン</dfn>とは...`
- **変換後（2回目以降）:** `<span id="idx-..." class="index-term" data-yomi="れすぽんしぶでざいん">レスポンシブデザイン</span>`

#### 2.2 セマンティックHTML（Ruby 4.0.0 の Set で実現）

```ruby
class Indexer
  def initialize
    # Ruby 4.0.0: require "set" 不要、Set リテラル構文を使用
    @seen_terms = Set[]
    @index_data = Hash.new { |h, k| h[k] = Set[] }
  end
  
  def process_content(markdown_text, file_path)
    # [用語|読み] または [用語] 記法のパース
    markdown_text.gsub!(/\[([^|\]\n]+)(?:\|([^\]\n]+))?\](?!\()/) do |match|
      term = $1
      yomi = $2 || infer_yomi_with_mecab(term)  # 読みがなければ MeCab で推測
      
      # ID の生成（ハッシュベースで一意性を保証）
      id = "idx-#{term.hash.abs.to_s(36)}"
      
      # 索引データの蓄積（Set で重複を自動排除）
      @index_data[term] << { yomi: yomi, link: "#{file_path}##{id}" }
      
      # 初出かどうかの判定に Set を活用（O(1) の高速検索）
      if @seen_terms.include?(term)
        "<span id='#{id}' class='index-term'>#{term}</span>"
      else
        @seen_terms << term
        "<dfn id='#{id}' class='index-term'>#{term}</dfn>"
      end
    end
  end
  
  private
  
  # MeCab で読みを推測
  def infer_yomi_with_mecab(term)
    return term unless mecab_available?
    
    mecab = Natto::MeCab.new
    yomi_parts = []
    
    mecab.parse(term) do |node|
      features = node.feature.split(',')
      reading = features[7]  # カタカナ読み
      
      if reading && reading != '*'
        yomi_parts << katakana_to_hiragana(reading)
      else
        yomi_parts << node.surface
      end
    end
    
    yomi_parts.join('')
  end
  
  def katakana_to_hiragana(str)
    str.tr('ァ-ヶ', 'ぁ-ゖ')
  end
  
  def mecab_available?
    @mecab_available ||= begin
      require 'natto'
      true
    rescue LoadError
      false
    end
  end
end
```

**ポイント**:
- **初出箇所**: `<dfn>` タグを使用（HTML仕様に準拠）
- **2回目以降**: `<span class="index-term">` を使用
- **読み省略時**: MeCab で自動推測し、`vs index:match` で確認・修正
- Ruby 4.0.0 の Set で同一用語の出現順を高速に追跡し、自動で切り替え

### Step 3: 索引データの収集とソート
全原稿を走査し、以下のデータを集計・ソートします。
1. **収集:** `{ 用語, よみ, リンク(ID) }` のリストを作成。
2. **名寄せ:** 同じ用語が複数箇所にある場合、リンク先を配列にまとめる。
3. **ソート:** 「よみ」に基づき五十音順・アルファベット順に並べ替え。

### Step 4: 索引ページの生成
ソート済みデータから `index.html` を出力します。
```html
<ul class="index-list">
  <li>レスポンシブデザイン ‥‥ <a href="ch1.html#idx-1"></a></li>
  <li>イベントリスナ ‥‥ <a href="ch2.html#idx-2"></a>, <a href="ch3.html#idx-5"></a></li>
</ul>
```

### Step 5: CSSによるページ番号表示

Vivliostyle のレンダリング時に、`target-counter` 関数を用いてページ番号を自動挿入します。

```css
.index-list a::after {
  content: target-counter(attr(href), page);
}
```

## 3. 索引語の自動抽出とフィルタリング

単純な単語抽出ではノイズ（不要な語句）が増えるため、以下のロジックで「索引に載せるべき重要語」を判定します。

### 3.1 抽出ターゲット（構造的アプローチ）

以下の箇所に含まれる語句を優先的に抽出します。

- **見出し**（`#`, `##`）に含まれる名詞
- **強調**（`**...**`）やコードスパン（`` `...` ``）
- **`<dfn>` タグ**で囲まれた定義語
- **`[用語|読み]` 記法**で明示的にマークアップされた語句

### 3.2 解説箇所の特定（文脈的アプローチ）

単なる出現（ノイズ）と解説（索引対象）を区別するフィルタ。

#### A. 定義表現のパターンマッチング

日本語の技術解説に特有の言い回しを正規表現で検出：

- `「〜とは、」`
- `「〜を……と定義します」`
- `「〜という概念は、」`
- `「〜について解説します」`

これらの表現の直前にある単語は、解説ページである確率が非常に高い。

#### B. 初出箇所の優先

その章で初めて登場した箇所を候補とする。

#### C. TF-IDF 法による重要語判定

特定の章に集中して出現し、全章にまたがらない単語を「その章のキーワード」とみなす。

```
TF-IDF = TF（単語の出現頻度） × IDF（その単語が全章のうち何章に出現するか）
```

- **TF (Term Frequency)**: その章でたくさん出てくるほど重要
- **IDF (Inverse Document Frequency)**: 多くの章で共通して出てくる言葉（「本」や「作業」など）は重要度を下げる

#### D. 近傍語（Co-occurrence）解析

用語の近くに関連語が多く含まれている箇所は解説である可能性が高い。

- 例: 「HTML」の近くに「構造」「要素」「タグ」「言語」→ 解説の可能性大
- 逆に「書く」「開く」などの動詞が多い → 作業手順の可能性大

### 3.3 実装運用案（ハイブリッド方式）

1. **スキャン**: Ruby が自動で索引候補（YAML 等）を作成
2. **編集**: 著者がそのリストを確認し、不要なものを削除・修正
3. **生成**: 確定したリストに基づき索引をビルド

## 4. データ構造例 (Ruby 4.0.0 内部)

```ruby
# Ruby 4.0.0: Set を活用したデータ構造
{
  "名前空間" => Set[
    { yomi: "なまえくうかん", link: "chapter1.html#idx-101" },
    { yomi: "なまえくうかん", link: "chapter3.html#idx-305" }
  ],
  "配列" => Set[
    { yomi: "はいれつ", link: "chapter2.html#idx-202" }
  ]
}

# 索引ページ生成時のソートとグループ化
sorted_terms = @index_data.sort_by { |term, occurrences| occurrences.first[:yomi] }
groups = sorted_terms.group_by do |term, occurrences|
  determine_kana_row(occurrences.first[:yomi])
end

# groups の構造例
{
  "あ" => [["アクセサー", Set[...]], ...],
  "な" => [["名前空間", Set[...]], ...],
  "は" => [["配列", Set[...]], ...]
}
```

### 4.1 五十音順ソートと行判定

```ruby
def determine_kana_row(yomi)
  # 読みの先頭一文字から行を判定（正規表現を活用）
  case yomi[0]
  when /[あ-お]/ then "あ"
  when /[か-ご]/ then "か"
  when /[さ-ぞ]/ then "さ"
  when /[た-ど]/ then "た"
  when /[な-の]/ then "な"
  when /[は-ぽ]/ then "は"
  when /[ま-も]/ then "ま"
  when /[や-よ]/ then "や"
  when /[ら-ろ]/ then "ら"
  when /[わ-ん]/ then "わ"
  when /[a-e]/i then "A"
  when /[f-j]/i then "F"
  when /[k-o]/i then "K"
  when /[p-t]/i then "P"
  when /[u-z]/i then "U"
  else "その他"
  end
end
```
## 5. 形態素解析による名詞抽出

### 5.1 ツール選定

- **natto** または **mecab-ruby**（MeCab の Ruby バインディング）
- MeCab 辞書: mecab-ipadic-NEologd（新語・固有名詞に強い）

### 5.2 複合名詞の抽出

連続する名詞を結合して「複合名詞」として抽出：

```ruby
# 例: 「レスポンシブ」（名詞）＋「デザイン」（名詞）
# → 「レスポンシブデザイン」（複合名詞）
```

### 5.3 専門用語辞書との照合

技術書であれば、以下のデータセットと照合し、一致したものを候補に挙げる：

- Wikipedia のタイトル一覧
- 技術用語のデータセット（IT用語辞典など）
- プロジェクト固有の用語集（`config/technical_terms.yml` など）

これにより、無関係な一般名詞を排除できる。

## 7. prh を利用した辞書方式

`textlint` の辞書機能を利用して索引語の抽出とフィルタリングを行う案。

「あらかじめ登録された辞書 `terms.yml` に基づいて機械的にマッチングして処理する」**という prh（proofreading helper）アプローチ**は、技術書の索引作りにおいて実用的で、エンジニアの著者にとっても親しみやすい設計になる。

### 7.1 索引辞書 (terms.yml) の設計案

prh の形式に倣い、索引語とその読み、さらに「どういう条件で索引に含めるか」を定義できるようにする。

```yaml
# vivlio-starter 索引辞書案 (terms.yml)
rules:
  - pattern: /アクセサー?/
    yomi: あくせさー

  - pattern: /イベントリスナ[ー]?/
    yomi: いべんとりすな

  # 「HTML」のような頻出語の制御
  - pattern: /HTML/
    yomi: えいちてぃーえむえる
    # 「〜とは」という解説の時だけ自動抽出対象にする
    auto_index_pattern: /HTML(?=とは|について)/
```

### 7.2 prh 方式を索引に応用するメリット

この「辞書ベース」のアプローチを vivlio-starter に取り入れると、以下の課題を解決できる。

1. **表記揺れの吸収**  
   「アクセサ」と「アクセサー」を同じ索引語として集計できる。
2. **「解説」と「単なる語句」の区別**  
   例: 自動抽出用パターン `/(HTML|CSS)(?=とは|の基本)/` を使い、解説箇所のみ索引用にする。
3. **プログラマブルな管理**  
   「ソースから動的に長音ありリストを生成する」といった運用を Ruby 側で実装しやすい。

### 7.3 Ruby Gems 側での処理フロー

vivlio-starter がビルド時に行う処理を整理すると以下の通り。

1. **辞書の読み込み**: `terms.yml` をロード
2. **本文のスキャン**:
   - 各ルールを正規表現としてコンパイル
   - Markdown / HTML 内を検索
   - マッチした箇所に、prh のような置換ではなく **タグ付け（ID 付与）** を行う
3. **索引ページの構築**:
   - マッチした場所のファイル名と ID をリスト化
   - 五十音順（`yomi` 順）に並べ替えて `index.html` を生成


## 8. ハイブリッド案

「ものぐさな著者（あるいは忙しい著者）」にとっての利便性と索引の質を両立させるには、**構造抽出**と**辞書マッチ**を組み合わせたハイブリッド自動化が現実的。著者が書いた原稿から重要語を浮き彫りにするアプローチのほうが、その本にとって価値ある索引になりやすい。

### 8.1 自動抽出の 3 レイヤー

優先順位を定めた 3 レイヤー構成で語句を抽出する。

1. **レイヤー1: 構造的抽出（ほぼ確実な索引語）**
   - 見出し（`#`, `##`）内の名詞
   - `<dfn>` タグで定義される語句
   - 太字（`**...**`）やコードスパン（`` `...` ``）で強調された語句
2. **レイヤー2: 辞書マッチ（`terms.yml` / prh 方式）**
   - 重要語を辞書に登録しておくことで、読み（`yomi`）を含めて抽出
3. **レイヤー3: 文脈抽出（「HTMLとは」問題の解決）**
   - 定義パターン `/ (.+?)(?:とは、|は、?.*定義されます)/`
   - 出現頻度の偏り（特定章に偏って登場する語）を利用

### 8.2 スコアリングアルゴリズム

すべてを自動登録せず、**索引スコア**で閾値を超えた語だけ採用する。

| 条件                                         | スコア | 判定             |
|----------------------------------------------|:------:|------------------|
| 見出しに含まれている                         | +100   | 自動登録         |
| `<dfn>` タグで囲まれている                   | +100   | 自動登録         |
| `terms.yml` にあり、かつ「とは」が続く       |  +80   | 自動登録         |
| `terms.yml` にあるが、通常の出現のみ         |  +20   | 候補（要確認）   |
| 文中で 1 回だけ出てくる一般名詞             |   +5   | 無視             |

### 8.3 Ruby 実装の要: 2 パス・ビルド

vivlio-starter ではビルド工程を 2 段階に分けると整理しやすい。

1. **パス 1（解析）**
   - 全原稿をスキャンしスコアリング結果から索引候補リストを生成
   - ページ番号の代わりに「ファイル名 + アンカー ID」を保持
2. **パス 2（生成）**
   - 確定した索引用語から `index.html` を出力
   - 本文側では該当箇所に `<span id="idx-xxx">` を挿入して Vivliostyle へ渡す

### 8.4 著者向けユーティリティ

著者の作業負荷を下げる CLI フローの例。

1. **`vs index:match`**
   - 原稿を走査して `index-candidates.yml` を出力
   - `[用語]` 形式（読み省略）の読みを MeCab で自動推測
   - 読み間違いの可能性がある項目を警告表示

2. **著者が YAML を編集**
   - 不要語を `enabled: false` に設定
   - 読み間違いを修正（例: HTML → えいちてぃーえむえる）
   - 読みを調整（五十音順の微調整）

3. **`vs build`**
   - 編集済み YAML を元に索引つき書籍を一括生成
   - MeCab で推測された読みを使用
