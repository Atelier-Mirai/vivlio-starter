# EPUB/Kindle コードブロックの行番号と折返し 仕様（将来タスク）

> 作成日: 2026-06-20
> ステータス: **将来やりたいこと（一部未解決）**
> 対象: リフロー型 EPUB（クリーン EPUB / Kindle）でのソースコード表示。行番号と各行の対応、長行の折返し。
> 関連: `epub-kindle-target-split-spec.md`, `epub-kindle-layout-spec.md`（Kindle コードのテーブル化）, `stylesheets/code.css`, `stylesheets/prism.css`

---

## 0. 現状（2026-06-20）

### クリーン EPUB（Kobo / Apple Books）
- Prism の `pre.line-numbers`（絶対配置ガター）をそのまま使用。
- **是正済み（§6 A 案）**: `body.vs-epub pre[class*="language-"] { white-space: pre-wrap; overflow: visible }` を追加し、Apple Books の特定ウィンドウ幅でコードがクリップ（消失）する不具合を解消。全文表示される。
- **未解決**: ウィンドウ幅可変のリフローでは、長行が折り返されると行番号（固定行高で配置される `.line-numbers-rows`）と実際の行がずれる。短い行が大半のコードでは目立たないが、長行が多いと番号と内容が対応しなくなる（`epub_codes_v3.png`）。

### Kindle（KFX・テーブル方式）
- `convert_code_blocks_for_epub!` が Prism 行番号コードを 2 列テーブル（`vs-code-epub`: 番号セル｜コードセル）へ変換。各論理行が `<tr>` なので**番号↔行の対応は理屈上崩れない**。
- **不具合（`kindle_codes.png`）**:
  1. **2 桁以上の行番号が縦に折り返る**（例: `10` が `1` / `0` の 2 行）。番号セル `td.vs-code-num { width: 2.5em; white-space: nowrap }` を指定しているが、Kindle KFX がテーブルセルの `width` / `white-space: nowrap` を十分に尊重せず、狭い番号列で数字が折り返るため。
  2. **折返し行と非折返し行の行間が不均一**。上記 1 により番号が 2 行になった行や、コードが折り返した行はセル高が増え、`vertical-align: middle` の番号が中央に浮く。結果「8→9 は 1 行分、10→11 は約 2 行分」のような不揃いが生じる。

## 1. 論点

リフロー EPUB で「行番号」と「コード行」を**確実に対応**させつつ**全文表示**する、決定版の方式を定める。クリーン EPUB（Apple Books 等・CSS 表現力が高い）と Kindle（KFX・CSS 制約が強い）で最適解が異なりうる。

## 2. 方式candidate（当初提案 A〜E）

| 案 | 内容 | 対応崩れ | 利点 | 欠点 |
|---|---|---|---|---|
| **A. コードもテーブル化** | クリーン EPUB も Kindle と同じ `vs-code-epub`（番号セル｜コードセル） | 解消（行＝tr） | 対応が崩れない・改ページ安全・Kindle と統一 | Kindle の現行テーブル不具合（§0）を**先に直す必要**。Prism のガター見た目が変わる |
| **B. 行番号を非表示** | リフロー時は `.line-numbers-rows` を隠し gutter 解除 | 解消（番号無し） | 最シンプル | 行番号が無くなる（PDF にはある） |
| C. 番号をテキスト接頭辞に | 各行頭に番号を実テキストで埋め込み（ぶら下げインデント） | ほぼ解消 | 番号がコードと共に流れる | コピー時に番号混入・実装手間 |
| D. 固定レイアウト EPUB | fixed-layout で PDF 同様 | 完全一致 | 見た目完全再現 | リフロー不可・文字拡大不可・大容量・本単位 |
| E. 折返さず横スクロール | `white-space:pre; overflow-x:auto` | 一致 | 番号維持 | Apple Books は横スクロール非提供→クリップ再発 |

## 3. テーブル方式（A）を採る場合に必要な Kindle 不具合の修正

A 案で「番号↔行の対応」を根本解決するには、§0 の Kindle テーブル不具合を直す前提条件がある。検討する是正案（要実機検証）:

1. **番号の縦折返し回避**:
   - 番号セルの幅を広げる（`width` ではなく十分な `min-width`、例 3〜3.5em）。
   - 番号を `<span style="white-space:nowrap">` で包む、または `&#8239;`（narrow no-break space）等で分割を抑止。
   - それでも KFX が折り返す場合、番号セルを**画像化**するか、桁数に応じて幅を可変にする。
2. **行間の均一化**:
   - 番号セルを `vertical-align: top` にして、折返し行でも番号が 1 行目に揃うようにする。
   - 行（`tr` / セル）の余白を 0 に近づけ、`line-height` を本文と揃える。
   - 折返しは許容しつつ、番号は論理行に 1 つ・上揃えで対応を保つ。
3. クリーン EPUB（Apple Books）はテーブル描画が素直なため、A 案導入で §0 の desync は解消が見込める（要実機確認）。Kindle は上記 1・2 の調整が成否を分ける。

## 4. 推奨と進め方

- **第一候補: A（テーブル化）＋ Kindle テーブル不具合の修正**。番号↔行を根本解決でき、クリーン/ Kindle を統一できる。ただし Kindle 実機での反復調整（§3）が必須。
- **簡易策: B（行番号非表示）**。実装が軽く、desync が原理的に起きない。行番号が要らない本ならこれで十分。
- 当面（本タスク未着手の間）はクリーン EPUB は §0 の pre-wrap（全文表示・番号は長行でずれることがある）を維持、Kindle は現行テーブル（§0 の不具合あり）を維持する。

## 5. 受け入れ条件（実装時）

- クリーン EPUB（Apple Books）でウィンドウ幅を変えても全コードが表示され、行番号が各論理行に対応する（または B 採用なら番号無しで全文表示）。
- Kindle で 2 桁以上の行番号が縦折返しせず、折返し行・非折返し行の行間が均一。
- PDF のコード表示（行番号付き）は不変（EPUB 専用後処理／CSS で行う）。
