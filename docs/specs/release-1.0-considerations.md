## A. リリース判断に関わる大きめの論点

コードの細部とは別レイヤーの「意思決定」であり、後から覆りにくいため、
正式版の検討時にまず決めておきたい3点。

### A-1. Ruby 4.0 必須という前提（最大の検討点）

- 現状: gemspec `required_ruby_version = '>= 4.0'`
- 課題: Ruby 4.0 はごく最近のリリースで、利用者の多くはまだ 3.x。間口が大きく狭まる。
- 検討: `Data.define` やパターンマッチ程度であれば Ruby 3.2〜3.4 でも動作するはず。
  本当に 4.0 固有機能へ依存しているかをコード調査し、依存していなければ
  `>= 3.3` 等への緩和で対象ユーザーが大きく広がる。
- 決定すべきこと: **最新前提を貫くか／間口を広げるか**。
- 着手の推奨度: 高（影響範囲が広く、後戻りしにくい）。

### A-2. 外部バイナリ依存のオンボーディング体験

本 gem は内部で以下をシェル経由で利用する。これらは `gem install` では導入されない。

- Vivliostyle (npm)
- Ghostscript
- poppler (`pdfinfo`)
- qpdf
- MeCab (`natto` 経由・要 libmecab)

確認すべき点:

- `vs doctor` が**全依存を漏れなく検査**し、不足時に**インストール手順を提示**できているか。
- README「追加ツールのインストール」が上記を**網羅**しているか。
- 依存が無い場合に**クラッシュではなく親切なエラー**で停止するか。

CLI として最も「正式版らしさ」が表れる部分。

### A-3. RubyGems.org への公開可否

- 現状: homepage は GitHub を指す。
- 決定すべきこと: **rubygems.org へ push するか／GitHub からの直接インストール運用にするか**。
- 公開する場合の付随作業:
  - `rubygems_mfa_required` を `false` → **`true`** へ（公開 gem のセキュリティ既定）。
  - バージョンを `1.0.0-beta` → **`1.0.0`** に確定
    （`config/book.yml` の `project.version` は既に `1.0.0`。gem 側のみ beta が残存）。
