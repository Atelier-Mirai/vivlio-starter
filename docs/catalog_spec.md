# catalog.yml 仕様

`config/catalog.yml` は、本の「章構成カタログ」を定義するファイルです。

- どの章ファイル（basename）が本の一員か
- まえがき・本文・付録・あとがきの論理的な構成
- （任意で）初級編 / 中級編 / 第一部 などの部タイトル

を著者向けに宣言し、ビルド時にはそこからビルド対象の章一覧を導出します。

`book.yml: chapters` は廃止し、章構成のソース・オブ・トゥルースは `config/catalog.yml` に一本化します。

---

## 1. ファイルの場所

- パス: `config/catalog.yml`
- YAML 形式のテキストファイル
- 無い場合の後方互換は考えず、「あること」が前提

---

## 2. 基本構造

トップレベルのキーとして、以下を定義可能です。

- `PREFACE`    … まえがき相当（例: 02-preface）
- `CHAPTERS`   … 本文
- `APPENDICES` … 付録
- `POSTFACE`   … あとがき

### 2.1 もっとも単純な例

```yaml
PREFACE:
  - 02-preface

CHAPTERS:
  - 11-fixed-footer
  - 12-follow-menu
  - 13-mobile-menu
  - 14-muuri
  - 15-chart

APPENDICES:
  - 91-books
  - 92-links
  - 93-cheats

POSTFACE:
  - 98-postface
```

各要素は **basename（拡張子なし）** で記述し、実際のファイルは `contents/<basename>.md` として参照されます。

`catalog.yml` 上では、次のいずれの表記も許容されます。

- `11-fixed-footer`
- `11-fixed-footer.md`

内部的には、`.md` 拡張子は自動的に取り除かれ、いずれも同じ basename（ここでは `11-fixed-footer`）として扱われます。

`CHAPTERS` および `APPENDICES` では、従来の `book.yml: chapters` と同様に、次のような **番号・範囲指定のショートハンド** も許容します。

- `21`            … 章番号 21 の章（例: `21-foo.md`）
- `21-25`         … 章番号 21〜25 に対応する章
- `21-25, 38`     … 21〜25 と 38 の組み合わせ

これらのトークンはビルド時に展開され、実際の basename リストに変換されます（章番号は `contents/` 以下のファイル名から解決されます）。

---

## 3. 部タイトルによるグルーピング

`CHAPTERS`（および必要なら他のセクション）では、**部タイトル＋配下の章リスト**を表現できます。

```yaml
CHAPTERS:
  - 初級編:
      - 11-fixed-footer
      - 12-follow-menu
  - 中級編:
      - 13-mobile-menu
      - 14-muuri
  - 上級編:
      - 15-chart
```

- キー（ここでは `初級編`, `中級編`, `上級編`）は **著者向けのメタ情報** です。
- ビルド時は「部タイトル自体」は章としては扱わず、配下の basename をフラットに展開して利用します。
- 将来的に「第一部」と書かれた紙を 1 枚挟むなどの機能拡張を行う場合、このメタ情報を利用しますが、現状はフラット展開のみを行います。
- 同様の構造は、`PREFACE` / `APPENDICES` / `POSTFACE` に対しても許容しますが、**ビルド挙動には影響しません（メタ情報扱い）** です。

---

## 4. basename と章番号レンジ

各 basename は、先頭の 2 桁の数字から「章番号」を決定します。

- 例: `11-fixed-footer` → 章番号 11
- 例: `91-books`        → 章番号 91

ビルド内部では、章番号レンジを以下のように扱います（既存実装と同様）:

- `PREFACE_RANGE`  = `2..2`    （例: 02-preface）
- `MAIN_RANGE`     = `11..89`  （本文）
- `APPX_RANGE`     = `91..97`  （付録）
- `POSTFACE_RANGE` = `98..98`  （あとがき）

また、番号 `00` / `01` / `99` が `catalog.yml` に書かれている場合は、次のように解釈します。

- `00-*.md` … 実質的に `PREFACE` と同等の扱い
- `01-*.md` … 実質的に `CHAPTERS` と同等の扱い
- `99-*.md` … 実質的に `POSTFACE` と同等の扱い

**重要:**

- `catalog.yml` 上でどのセクションの下に書かれているかにかかわらず、
  ビルドロジックは「basename の章番号レンジ」を見て
  - まえがき
  - 本文
  - 付録
  - あとがき
  を判断します。
- `PREFACE` / `CHAPTERS` などのキーや部タイトルは、あくまで **著者向けのガイド** です。

---

## 5. 特殊ページと番号の扱い

従来は次のような basename を特別扱いしていました:

- `00-titlepage` （タイトルページ）
- `01-legalpage` （奥付に先立つリーガルページ）
- `99-colophon`  （奥付）

今後は、番号 `00` / `01` / `99` を通常の章としても使えるように **解放** します。

代わりに、これらの特殊ページは **内部用 basename** として扱います。

- `contents/_titlepage.md`  … タイトルページ
- `contents/_legalpage.md`  … リーガルページ
- `contents/_colophon.md`   … 奥付

`catalog.yml` には原則としてこれらを記述せず、システム内部で利用されるページとして扱います。

中間 PDF のファイル名も、番号ではなく内部名ベースに変更します。

- `_titlepage_legalpage.pdf` … 旧 `00-01-front.pdf`
- `_preface_toc.pdf`         … 旧 `02-03-front.pdf`
- `_colophon.pdf`            … 旧 `99-colophon.pdf`

これらは最終的に `output.pdf`（およびリネーム後の最終出力）へマージされる中間成果物であり、
章番号レンジからは切り離されます。

## 6. `vs build` とサブセット指定

`book.yml: chapters` は廃止し、サブセット指定は **CLI でのみ** 行います。

### 6.1 基本

```bash
vs build          # catalog.yml に定義された全章をビルド
vs build 11-13    # 章番号 11〜13 に該当する basename のみビルド
vs build 02,11,98 # 章番号 2, 11, 98 に該当する basename のみビルド
```

### 6.2 トークンの解釈

`vs build` の引数は、次のように解釈されます。

- `N`     … 章番号 N（例: `11`）
- `N-M`   … 章番号 N〜M の範囲（例: `11-13`）
- カンマ区切り: `11,13-15,91`

### 6.3 ビルド対象の決定と CLI 優先

ビルドロジックは、CLI 引数の有無によって挙動が変わります。

- **CLI 引数なし (`vs build` のみ)**  
  - 1. `catalog.yml` から全 basename を収集
  - 2. 各 basename から章番号を抽出
  - 3. 存在しない章ファイル（`contents/<basename>.md` が無いもの）をスキップしつつ警告を出す
  - 4. 残った章をすべてビルド対象とする

- **CLI 引数あり（例: `vs build 21-25`）**  
  - 1. CLI 引数から章番号集合を展開（`21-25` → 21,22,23,24,25 など）
  - 2. 章番号に対応する `contents/` 以下のファイルを探索し、存在するものだけをビルド対象とする
  - 3. 対応するファイルが存在しない章番号（例: `23-*.md` が無いなど）は、警告を出しつつスキップする

このとき、**CLI 引数が優先**されます。`catalog.yml` 側に 11-15, 31-35 のようなショートハンドだけが書かれていても、
`vs build 21-25` と指定された場合は 21〜25 に該当する章だけをビルドします（カタログに含まれていなくても構いません）。

### 6.4 catalog.yml 側でのサブセット

- 特定の版だけ章を減らしたい場合などは、`catalog.yml` 上で **行をコメントアウト** することで対応できます。
- `catalog.yml` に存在しない章番号を `vs build` の引数に指定した場合は、単に無視されます（エラーにはなりません）。

## 7. バリデーションとエラー

実装として、次のようなバリデーションを行うことが推奨されます。

- **catalog.yml のパースエラー**  
  `catalog.yml` 自体が不正な YAML でパースに失敗した場合は、その時点でエラーとしてビルドを中止する。
- **重複章番号**  
  同じ章番号を持つ basename が複数存在する場合はエラーとし、ビルドを中止する。
- **不整合な catalog.yml**  
  章番号レンジに明らかに合わない basename が、意図しないセクションに書かれている場合
  - 例: `91-books`（付録）が `CHAPTERS` に書かれている
  - → ビルドは通しつつ、警告ログを出力する（メタ情報の不整合として扱う）。
 - **存在しない章ファイル**  
   `catalog.yml` に記載された basename に対応する `contents/<basename>.md` が存在しない場合は、該当章をスキップしつつ警告ログを出力する。ビルド自体は続行し、その他の章については通常どおり処理を行う。
 - **ショートハンドと basename の重複**  
   例: 
   ```yaml
   CHAPTERS:
     - 21-25
     - 23-foo
   ```
   のように、範囲指定により含まれる章が個別の basename としても列挙されている場合、
   ビルド対象としては重複を単に `uniq` して処理しつつ、重複がある旨を警告ログとして出力する。

## 8. 将来拡張のためのメタ情報

`catalog.yml` の部タイトルやセクション分割は、将来的に次のような用途に使える余地があります。

- 「初級編だけを別冊にする」など、部単位のビルドバリアント
- 目次や PDF アウトラインに「第 I 部」「第 II 部」などの部見出しエントリを挿入する
- 特定の部ごとにテーマやレイアウトを切り替える

現時点では **ビルド挙動には影響しないメタ情報** としつつ、
この仕様に沿って `catalog.yml` を維持しておくことで、後から拡張しやすい構造になっています。

## 9. 章操作コマンドとの連携

`vs create`, `vs delete`, `vs rename`, `vs renumber` などの **章操作コマンド** は、
章ファイル（`contents/*.md`）に対する変更と同時に `catalog.yml` も更新し、整合性を保つものとします。

- `vs create` で新しい章（例: `21-new-chapter.md`）を作成した場合、章番号に応じて次の規則でセクションを決定し、basename を追記する。
  - 0（`00-*.md`）        → `PREFACE`
  - 1〜89                 → `CHAPTERS`
  - 91〜98                → `APPENDICES`
  - 99（`99-*.md`）       → `POSTFACE`
  ユーザーが手動で `catalog.yml` を編集して別セクションへ移すことも可能だが、
  通常はコマンド経由での更新を推奨する（マニュアルで注意喚起する想定）。
  
  `CHAPTERS` が部タイトルで分かれている場合（例: `初級編:` / `中級編:` など）、
  新しい章は **章番号が最も近い既存の章（若い番号を優先）を含む部タイトルの直後** に挿入します。  
  適切な部が見つからない場合は、`CHAPTERS:` 直下の末尾に追記します。
- `vs delete` で章を削除した場合、`catalog.yml` からも該当エントリを削除する。
- `vs rename` で basename を変更した場合、`catalog.yml` 内の同一 basename も新しい名前に置き換える。
- `vs renumber` で章番号を変更した場合、ファイル名の変更に加え、`catalog.yml` 内の該当 basename も同様に更新する。
  - 章番号レンジが変わる場合（例: 11 → 91）、セクションも自動的に移動する。  
    例: 
    ```yaml
    CHAPTERS:
      - 11-fixed-footer
    APPENDICES:
    ```
    を renumber で 11→91 に変更した場合、
    ```yaml
    CHAPTERS:
    APPENDICES:
      - 91-fixed-footer
    ```
    のように `APPENDICES` 側へ移動する。

`catalog.yml` に番号・範囲指定（`21-25, 38` など）が含まれている場合、
これらのコマンドが更新を行う際には、**ショートハンドはフラットな basename リストに展開して書き戻す** 方針とします。

```yaml
CHAPTERS:
  - 21-25
``` 

のような記述は、更新後には例えば次のようになります。

```yaml
CHAPTERS:
  - 21-fixed-footer
  - 22-...
  - 23-...
  - 24-...
  - 25-...
```

このとき、`CHAPTERS:` や、ユーザーが明示的に指定した部タイトル（例: `初級編:`）といった
上位の構造はそのまま尊重して残します。
