# 箇条書き（ネスト・マーカー種別）拡張の検討メモ

> 作成日: 2026-07-12
> ステータス: **確定仕様へ昇格済み（2026-07-12）** — 実装調査を経て `nested-list-notation-spec.md` へ昇格した。本メモは検討経緯の記録として残す。
> 対象: 番号付き/番号なし箇条書きの入れ子、マーカー種別（A./（A）/a./i. 等）
> 関連: `stylesheets/chapter-common.css`（現行の ol/ul スタイル、208〜218行目）, `docs/specs/at-directive-ideas.md`（`:::{.class}` を優先する既存方針）

## 0. 前提・方針

**vivlio-starter 独自記法にはしない**。既存の Markdown 方言（特に Pandoc）が持つ枯れた記法を優先して採用し、方言でカバーできない部分だけ、このプロジェクト既存の `:::{.class}` ブロック装飾（`:::{.notice}` 等と同じ扱い）に乗せる。新しい sigil（`@` 等）は導入しない。

現状（2026-07-12 時点）: `ol`/`ul` は完全に CommonMark 標準のまま。マーカー種別（英字/ローマ数字）や複合カウンタ表示のCSSは一切ない（`chapter-common.css:208-218` はマージン調整のみ）。

## 1. 単純な入れ子（番号なし/番号ありとも）

```
* 概要
  * この機能について
  * インストール方法
    * Macの場合
```

これは **CommonMark 標準の字下げネストそのまま**で実現できる。新しい記法は不要。VFM が CommonMark 準拠である限り、HTML構造（`<ul><li>…<ul>…</ul></li></ul>`）はすでに組めるはず。番号なしのマーカー変化（disc→circle→square 等の階層による見た目変化）もCSSのデフォルト挙動の範囲で対応可能、記法は増えない。

**要確認**: 現状これが崩れる/意図通りにならないケースがあれば、それは記法拡張の話ではなく前処理（pre_process）側の不具合の可能性があるため、別途切り分けが必要。

## 2. マーカー種別（A. / （A） / a. / i. / I. など）

CommonMark の順序リスト記法はアラビア数字の `1.`（または `1)`）のみを規定しており、英字・ローマ数字・括弧付きなどのマーカーは範囲外。何らかの合図が必要。

**採用案: Pandoc の `fancy_lists` 拡張をそのまま踏襲。**
「最初の項目のマーカー表記でそのリスト全体の様式が決まる」という Pandoc 既知のルール。

```
A. 手順その一
B. 手順その二
```
```
(a) 選択肢イ
(b) 選択肢ロ
```
```
i. 項目
ii. 項目
```

著者は「そう書けばそう見える」だけで済み、`{.class}` や独自 sigil を覚える必要がない。Pandoc 既知の方言であるため「独自記法」感が薄いのが利点。

**実装イメージ（fable 検討用メモ）**: マーカー文字列から様式を判定し、CSS `list-style-type`（`upper-alpha` / `lower-alpha` / `upper-roman` / `lower-roman`）または `list-style: none` + `::before` + `counter()`（括弧付きなど CSS 標準の `list-style-type` だけでは表現できない形）で描画。

**Kindle考慮（要検証）**: `::marker` の `content` カスタムや `::before` 生成マーカーが KFX でどこまで効くか未検証。既存の admonition ラベル（`EpubBuilder::ADMONITION_LABELS`）と同様、Kindle 向けには本物の文字を `<li>` 内に注入するフォールバックが要る可能性がある。

## 3. ネストした番号リストの表示（複合番号 vs 独立採番）

同じネスト構造でも、著者によって期待する見た目が割れる:

- **複合番号**: `1 / 1.1 1.2 / 2 / 2.1 2.2`（JIS的な日本語技術文書でよく見る形）
- **独立採番**: `1 / 1 2 / 2 / 1 2`（各階層が親と無関係に 1 から数え直す）

両者は**ソーステキストが同一**（どちらの階層も単に `1. 2. 3.` と書くだけ）になるため、マーカー文字からは判別不可能。著者の意図を伝える合図が必須。

**他ツールの前例**: Pandoc・LaTeX（`enumitem`）・素の HTML/CSS `ol` ネストは、いずれもデフォルトが「独立採番」。複合番号を出すには LaTeX ですら `label={1.\arabic*}` 等の明示指定が要る。

**採用案**:
- **デフォルトは独立採番（1/2 方式）** — 何も書かなければ普通のネストリストとして振る舞う。他ツールのプレビューと食い違わない、最も驚きの少ない挙動。
- **複合番号（1.1/1.2）が欲しい著者だけ `:::{.class}` で明示的に囲む** — 新しい記法を増やさず、既存の「素の Markdown で書けないものは `:::{.class}`」の流儀に乗せる。クラス名は要検討（例: `outline-numbers` 等、fable 側で最終決定）。

## 4. マーカー種別を階層ごとに変える書き方（`1 / (a) / (b)`）

```
1. 概要
   (a) 選択肢イ
   (b) 選択肢ロ
```

これは **§2 の副産物として自然に解決される**。子リストの先頭マーカーが文字通り `(a)` であるため、そのレベルは「マーカー文字列でスタイルが決まる」ルール（§2）によって自動的に「括弧付きアルファベット」と解釈される。§2 とは別に追加設計は不要——深さごとに違うマーカーを書き分けたい著者は、そのまま書けば意図が伝わる。

## 5. まとめ（振り分け表）

| やりたいこと | 対応方針 | 新規記法の要否 |
|---|---|---|
| 単純な入れ子（番号なし/あり） | CommonMark標準の字下げネストのまま | 不要 |
| マーカー種別変更（A./（A）/a./i./I.） | Pandoc `fancy_lists` 方式（先頭マーカーで様式判定） | 不要（Pandoc互換記法） |
| 階層ごとに違うマーカー（1/(a)/(b)） | §2 の副産物として自動解決 | 不要 |
| ネスト時の複合番号（1.1/1.2） | デフォルトは独立採番、複合番号は `:::{.class}` で明示 | `:::{.class}` のみ（新sigilなし） |
